# HobbyForge — Full Development Roadmap for Claude

## 0. Product Summary

**HobbyForge** is a personal desktop app for managing a Warhammer hobby collection. The app is focused primarily on painting, collecting, rules, army readiness, and gradually learning to play.

## 1. Core Product Vision

The app should become a personal Warhammer command center with four main pillars:

1. **Collection management**

   * Track owned models, units, factions, quantities, status, purchase info, photos, and notes.

2. **Painting progress and recipes**

   * Track the painting status of each unit/model.
   * Store painting plans, paint recipes, photos, basing ideas, tutorials, and lessons learned.

3. **Army readiness and list planning**

   * Build simple army lists from the collection.
   * Manually enter points.
   * See whether a list is assembled, painted, based, and ready to play.
   * Rules of the units


4. **Learning to play**

   * Store personal unit notes, roles, synergies, strategy ideas, battle logs, and lessons learned.

The long-term goal is not to build a competitive list optimizer first. The priority is to help the user go from “collector/painter” to “ready-to-play hobbyist”.

# 2. Recommended Technical Stack

## 2.1 Desktop App Framework

Use:

* **Tauri** for desktop packaging
* **React** for frontend
* **TypeScript** for type safety
* **Tailwind CSS** for styling
* **shadcn/ui** for UI components
* **SQLite** for local storage
* **Prisma** as ORM if compatible with the chosen Tauri setup

Alternative if Prisma creates too much friction inside Tauri:

* SQLite directly via a Tauri Rust plugin or a lightweight TypeScript-friendly SQLite wrapper.

## 2.2 Data Storage

Data should be local-first.

Recommended structure:

* SQLite database for structured data
* Local filesystem folder for images
* JSON export/import for backup

Suggested app data folder:

```text
HobbyForge/
├── database/
│   └── hobbyforge.sqlite
├── images/
│   ├── units/
│   ├── models/
│   ├── painting-progress/
│   └── references/
├── exports/
└── backups/
```

## 2.3 Design Style

The app should feel like a serious but enjoyable hobby dashboard.

Style direction:

* Clean desktop interface
* Dark mode first, but light mode possible later
* Strong use of cards, tags, status indicators, progress bars
* Avoid toy-like UI
* Make it feel like a personal command center

Suggested visual identity:

* Dark slate background
* Subtle faction-colored accents
* Rounded cards
* Compact tables for collection data
* Large visual tiles for painting projects

# 3. MVP Scope

The MVP should be deliberately focused.

## MVP Goal

The user can:

1. Add factions and units.
2. Track collection and painting status.
3. Create painting recipes.
4. Track paint inventory.
5. Build a simple army list manually.
6. See dashboard statistics.
7. Save personal play notes.

## MVP Must-Have Features

### 3.1 Dashboard

The dashboard should show:

* Total armies/factions tracked
* Total units owned
* Total models owned
* Painting completion percentage
* Assembly completion percentage
* Basing completion percentage
* Current active painting projects
* Recently updated units
* Army readiness summary

Example dashboard cards:

* “Tau Empire — 1,250 pts owned — 720 pts painted”
* “Ultramarines — 850 pts owned — 400 pts painted”
* “Current project: Tau Fire Warriors — Basecoat stage”

### 3.2 Collection Management

The user should be able to create, edit, delete, and view units.

Each unit should include:

* Faction
* Unit name
* Unit type/category
* Number of models
* Owned quantity
* Assembled status
* Primed status
* Painted status
* Based status
* Varnished status
* Points value manually entered
* Purchase date
* Purchase price
* Storage location
* Notes
* Main image

Suggested unit categories:

* HQ / Leader
* Battleline
* Infantry
* Elite
* Vehicle
* Monster
* Transport
* Character
* Dedicated Transport
* Other

For Warhammer 40K 10th edition, official categories may evolve, so allow user-defined categories.

### 3.3 Painting Status System

Each unit/model should have a painting progress status.

Suggested status values:

* Not started
* Built
* Primed
* Basecoated
* Shaded
* Layered
* Highlighted
* Details done
* Based
* Varnished
* Completed

Also support a manual percentage completion field.

Each unit should have:

* Painting status
* Painting percentage
* Active/inactive project flag
* Painting priority
* Target completion date

### 3.4 Painting Recipes

The user should be able to create painting recipes.

Each recipe should include:

* Recipe name
* Faction / army association
* Unit association optional
* Surface/area, e.g. armor, weapon, skin, cloth, lens, metal, base
* Primer
* Basecoat
* Shade/wash
* Layer
* Highlight
* Glaze/filter
* Weathering
* Technical paints
* Basing recipe
* Notes
* Tutorial links

Example recipe:

```text
Recipe: Tau White Armor
Area: Armor panels
Primer: White Scar / Grey Seer
Basecoat: Corax White or Ulthuan Grey
Shade: Apothecary White / thinned Soulblight Grey
Layer: Ulthuan Grey
Highlight: White Scar
Notes: Keep recess shading very controlled to avoid dirty white armor.
```

### 3.5 Paint Inventory

The user should be able to track paints.

Paint fields:

* Brand
* Paint name
* Paint type
* Color family
* Owned status
* Quantity
* Running low flag
* Wishlist flag
* Notes

Paint types:

* Primer
* Base
* Layer
* Contrast
* Shade
* Technical
* Dry
* Air
* Metallic
* Ink
* Oil
* Enamel
* Pigment
* Other

### 3.6 Army List Builder — Simple Version

The MVP should not attempt to validate every official list-building rule.

Core functionality:

* Create army list
* Select faction
* Add units from the user’s collection
* Manually enter points per unit
* Total points calculated automatically
* Show number of models
* Show painted/assembled/based status per unit
* Add tags: Casual, Learning, Narrative, Competitive, Test
* Add notes

Useful calculated fields:

* Total points
* Painted points
* Unpainted points
* Percentage painted by points
* Total models
* Battle-ready status

Example:

```text
List: Tau 1000 pts Learning List
Total: 995 pts
Painted: 620 pts
Ready: 62%
Unpainted blocker: Crisis Battlesuits
```

### 3.7 Strategy and Learning Notes

For each unit, allow personal tactical notes.

Fields:

* Battlefield role
* Strengths
* Weaknesses
* Best targets
* Synergies
* Mistakes to avoid
* Rules references
* Personal notes

Important: store only user-written notes and references. Do not scrape or reproduce copyrighted rules.

### 3.8 Battle Log

The user should be able to record games.

Fields:

* Date
* Opponent
* Opponent faction
* User faction
* Army list used
* Mission
* Points played
* Result: win/loss/draw
* Score
* MVP unit
* Underperforming unit
* Key lessons
* What to change next time
* General notes

MVP version can be basic, but the data model should support future analytics.

* Build incrementally.
* Do not rewrite the whole app unless necessary.
* Keep code modular.

# 21. Recommended MVP Cut Line

4. Create painting recipes.
5. Track paints.
