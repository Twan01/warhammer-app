<task type="auto">
  <name>Task 1: Write production and dev .bat launchers</name>
  <files>launch.bat, launch-dev.bat</files>
  <action>
    Create two Windows batch files at the repository root.

    **`launch.bat`** — production launcher. Behavior:
    1. Use `%~dp0` to resolve paths relative to the .bat file location (so double-click from File Explorer works regardless of current dir).
    2. Set a variable `EXE` to `%~dp0src-tauri\target\release\hobbyforge-scaffold.exe`.
    3. If `%EXE%` exists, launch it via `start "" "%EXE%"` and exit immediately (no lingering cmd window). Use `start ""` with the empty title — required so cmd doesn't try to interpret the quoted path as a window title.
    4. If `%EXE%` does NOT exist, print a friendly message ("HobbyForge isn't built yet — building now, this takes a few minutes...") and run `pnpm tauri build`, then re-check and launch.
    5. If `pnpm` is not on PATH, print a clear error ("pnpm not found. Install it from https://pnpm.io or run launch-dev.bat instead.") and `pause` so the user can read the message.

    Use `@echo off` at the top. Use `setlocal` to avoid leaking variables. Use `endlocal` before exit branches.

    **`launch-dev.bat`** — dev launcher. Behavior:
    1. `@echo off`, `cd /d %~dp0` (cd to repo root using the .bat's own directory).
    2. Print a one-line banner: "Starting HobbyForge dev server (Ctrl+C to stop)..."
    3. Run `pnpm tauri dev`. Do NOT use `start` — the dev server output should stream into this window so the user can see compile errors.
    4. After it exits, `pause` so the window stays open if there's an error to read.

    Why `start "" "%EXE%"` and not just `"%EXE%"`: directly invoking the exe from cmd would block the cmd window until the app closes. `start` detaches it. The `""` is an empty title required when the path is quoted.

    Why pnpm (not npm or yarn): tauri.conf.json's beforeDevCommand is "pnpm dev" and the repo has pnpm-lock.yaml. Using npm would either fail or create a competing lockfile.

    Why fall back to `pnpm tauri build` (production-grade) instead of `pnpm tauri dev` in launch.bat: the user asked for a launcher to OPEN THE APP, not to start a dev server. A built exe starts in ~1 second; `tauri dev` takes 10-30s and requires Node + Vite + Cargo all running.
  </action>

<task type="auto">
  <name>Task 2: Write PowerShell desktop shortcut creator + update .gitignore</name>
  <files>scripts/create-desktop-shortcut.ps1, .gitignore</files>
  <action>
    **`scripts/create-desktop-shortcut.ps1`** — one-time helper that creates a Desktop shortcut to `launch.bat` with the HobbyForge icon. Behavior:
    1. Resolve repo root: `$repoRoot = Split-Path -Parent $PSScriptRoot` (since the script lives in `scripts/`, parent = repo root).
    2. Build paths:
       - Target: `$repoRoot\launch.bat`
       - Icon: `$repoRoot\src-tauri\icons\icon.ico`
       - Shortcut: `[Environment]::GetFolderPath('Desktop') + '\HobbyForge.lnk'`
    3. Validate target exists; if not, `Write-Error "launch.bat not found at $target — run from a fully cloned repo."` and exit 1.
    4. Create the shortcut using the standard COM pattern:
       ```powershell
       $shell = New-Object -ComObject WScript.Shell
       $shortcut = $shell.CreateShortcut($shortcutPath)
       $shortcut.TargetPath = $target
       $shortcut.WorkingDirectory = $repoRoot
       $shortcut.IconLocation = $icon
       $shortcut.Description = "Launch HobbyForge"
       $shortcut.Save()
       ```
    5. Print success: `"Shortcut created at $shortcutPath"`.

    Add a comment header: `# Run once: powershell -ExecutionPolicy Bypass -File scripts\create-desktop-shortcut.ps1`

    Use `Set-StrictMode -Version Latest` and `$ErrorActionPreference = 'Stop'` for safety.

    **`.gitignore`** — append (do NOT clobber existing rules; read first, append):
    ```
    # User-local launcher overrides (optional - uncomment to ignore)
    # launch-local.bat
    # launch-dev-local.bat
    ```
    Why commented-out: the canonical launchers ARE checked in (so anyone cloning gets them). The block just documents how a user can create personal variants without dirtying git.

    Why a separate PowerShell helper instead of a third .bat: creating Windows shortcuts (.lnk files) requires the WScript.Shell COM object, which is awkward in pure cmd. PowerShell handles this in 6 lines. The user runs it ONCE.

    Why `WorkingDirectory = $repoRoot`: ensures `%~dp0` inside launch.bat resolves correctly when the shortcut is invoked from anywhere.
