## v0.2.13 Requirements — Data Integrity, Diagnostics & Product Coherence

### Data Integrity

- [x] **DI-01**: Applied recipe step progress is keyed by recipe_step_id (not order_index) — reordering steps does not move completion — *Phase 74, validated*
- [x] **DI-02**: Existing progress rows are migrated safely from order_index to recipe_step_id with section-disambiguated back-fill — *Phase 74, validated*
- [x] **DI-03**: Recipe metadata, sections, and steps save atomically in a single transaction — partial saves are impossible — *Phase 75, validated*
- [x] **DI-04**: Recipe graph save preserves existing section/step IDs (non-destructive, same as current five-phase diff) — *Phase 75, validated*
- [x] **DI-05**: package.json and tauri.conf.json versions match, enforced by a `pnpm check:version` script — *Phase 73, validated*

### Points & Validation

- [x] **PV-01**: All army list / Game Day / validation surfaces use a single centralized points resolver function — *Phase 76, validated*
- [x] **PV-02**: Points source is labeled in UI (e.g. "95 pts · synced", "100 pts · manual override", "unknown · needs review") — *Phase 76, validated*
- [x] **PV-03**: User can see whether a unit's point value was auto-matched or manually confirmed — *Phase 76, validated*
- [x] **PV-04**: User can manually confirm or override the unit-to-rules mapping — *Phase 76, validated*
- [x] **PV-05**: Duplicate or ambiguous unit-to-rules matches are flagged — *Phase 76, validated*
- [x] **PV-06**: List-level warnings (points exceeded, stale source) are shown once in summary panel, not repeated per unit — *Phase 76, validated*
- [x] **PV-07**: Unit-level warnings remain attached to relevant unit rows — *Phase 76, validated*

### Diagnostics

- [x] **DX-01**: Data Health page shows app version, schema versions, last sync date, and sync error count — *Phase 77, validated*
- [x] **DX-02**: Data Health page shows row counts for key tables (units, recipes, assignments, progress, points) — *Phase 77, validated*
- [x] **DX-03**: Data Health page flags orphaned progress rows, ambiguous point matches, and stale sync data — *Phase 77, validated*
- [x] **DX-04**: Data Health page runs diagnostics without blocking UI (async/lazy) — *Phase 77, validated*

### Backup

- [x] **BK-01**: User can create a backup of hobbyforge.db from the UI via file picker — *Phase 77, validated*
- [x] **BK-02**: Backup uses VACUUM INTO for safe SQLite copy (not raw file copy) — *Phase 77, validated*
- [x] **BK-03**: Backup status and last backup date are displayed on Data Health page — *Phase 77, validated*

### Dashboard

- [x] **DB-01**: Dashboard shows "Next Painting Action" driven by applied recipe progress (step description, time estimate, paint availability) — *Phase 78, validated*
- [x] **DB-02**: Dashboard shows "Ready to Play" summary driven by army list validation (points, unpainted count, sync freshness) — *Phase 78, validated*
- [x] **DB-03**: Dashboard shows "Data Health" summary (sync age, warnings count, backup status) — *Phase 78, validated*

### Game Day

- [x] **GD-01**: Game Day has an "End Game" action that creates a battle log pre-filled with army list, date, and opponent — *Phase 78, validated*
- [x] **GD-02**: User can record post-game learnings (what rules were forgotten, MVP/underperformer units) — *Phase 78, validated*
- [x] **GD-03**: Forgotten rules from after-action can become future Game Day reminders — *Phase 78, validated*
- [x] **GD-04**: Unit/list notes can be updated from the Game Day after-action flow — *Phase 78, validated*
