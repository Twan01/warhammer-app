## Overview

HobbyForge v2.0 adds three features that complete the "ready-to-play" workflow: a dedicated Paint Inventory page, an Army List Builder, and the Unit Playbook (personal stats block and strategy notes per unit). Phase 6 is a pure back-end foundation — schema migration, TypeScript types, all query and hook modules for all three features — so that Phases 7–9 build the UI on a verified data layer with zero migration risk.

### Phase 8: Army List Builder

**Goal:** Users can create and manage army lists drawn from their collection — adding and removing units, entering per-unit points overrides, and seeing auto-calculated totals (total points, painted points, battle-ready %) — and the unit delete flow warns before removing a unit that belongs to an active list

**Requirements:** ARMY-01, ARMY-02, ARMY-03, ARMY-04, ARMY-05, ARMY-06, ARMY-07

**Success Criteria:**
1. User can create an army list with name, faction, list type tag (Casual, Learning, Narrative, Competitive, Test), and notes
2. User can add units from their collection to a list and remove them — each unit shows painting status badge and assembled status
3. User can enter a per-unit points override; leaving it blank falls back to `unit.points`; effective points computed via `COALESCE(points_override, unit.points, 0)` in SQL

## Key Architecture Decisions (v2.0)

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| Phase 6 pure back-end foundation | Decouple migration risk from UI work; phases 7-9 build on verified data layer | Zero migration issues across phases 7-9 |
| COALESCE in SQL (not JS) for effective_points | SQL-side computation ensures consistency; never reimplemented in 3 places | Clean ArmyListSummaryBar: just sums `effective_points` |
| Full-replacement UPDATE for army_list_units | NULL must be passable to clear points_override back to inherited | Pitfall 2 avoidance; confirmed correct in smoke test |
| Zustand for paint inventory filters | Ephemeral UX — filters reset on navigation; no URL complexity | Consistent with Phase 3 collection filters pattern |
| Sibling portal for UnitPickerDialog | Radix portals nested inside Sheets cause z-index/context issues | Dialog + Sheet both visible simultaneously; confirmed Pitfall 1 |
| PlaybookTab SheetHeader/Footer outside Tabs | Edit/Delete must work from Playbook tab without closing sheet | Pitfall 5 avoidance; SheetContent overflow-y-auto retained |
| `key={unit?.id}` on SheetContent | Free PlaybookTab state reset on unit switch | Pitfall 6 avoidance (same pattern as ArmyListSheet Pitfall 6) |
