**Section 2: Sync Data Flow (AUDIT-02)**

Document the complete 5-step flow. For each step, include the exact file + line range reference and what happens:

Step 1 — CSV Fetch (src/hooks/useRulesSync.ts):
- `Promise.all` of 12 `fetchCsv()` calls via `@tauri-apps/plugin-http`
- Base URL: `https://wahapedia.ru/wh40k10ed/{filename}`
- Files: Factions.csv, Source.csv, Datasheets.csv, Datasheets_models.csv, Datasheets_abilities.csv, Datasheets_keywords.csv, Datasheets_wargear.csv, Abilities.csv, Stratagems.csv, Detachments.csv, Detachment_abilities.csv, Last_update.csv
- Custom User-Agent header to avoid Wahapedia bot blocking

Step 2 — Parsing + HTML stripping (src/hooks/useRulesSync.ts):
- `parseWahapediaCsv()` converts each raw CSV string to `Record<string,string>[]`
- `stripHtml()` applied to: damaged_description (datasheets), name+description (datasheet abilities), description (wargear), description (shared abilities), description+legend (stratagems), description+legend (detachment abilities)
- `parseLastUpdate()` extracts wahapediaVersion from Last_update.csv line 2

Step 3 — Tauri IPC (src/hooks/useRulesSync.ts -> src-tauri/src/lib.rs):
- `invoke("bulk_sync_rules", { payload: BulkSyncPayload })`
- BulkSyncPayload is a Rust struct with 13 fields: factions, sources, datasheets, models, abilities, keywords, wargear, shared_abilities, stratagems, detachments, detachment_abilities, last_sync_at, wahapedia_version
- Serialized as JSON over the Tauri IPC bridge

Step 4 — Rust transaction (src-tauri/src/lib.rs `bulk_sync_rules`):
- Opens direct sqlx connection (NOT plugin pool) for real transactions
- SqliteConnectOptions: WAL mode, 30s busy_timeout, create_if_missing=false
- `PRAGMA foreign_keys = OFF` — allows DELETE in any order
- `BEGIN TRANSACTION`
- DELETE all 11 rw_* tables (keywords, abilities, models, wargear, datasheets, sources, factions, rw_abilities, stratagems, detachment_abilities, detachments)
- INSERT loops for all 11 tables (some use INSERT OR IGNORE, keywords use plain INSERT)
- `INSERT OR REPLACE INTO rw_sync_meta` (singleton row, id=1)
- `COMMIT`

Step 5 — Response + Cache Invalidation (src/hooks/useRulesSync.ts):
- `bulk_sync_rules` returns `Result<(), String>` — **no row counts returned from Rust**
- TypeScript constructs `rowCounts` from local array `.length` (9 tables counted: factions, sources, datasheets, models, abilities, keywords, wargear, stratagems, detachments)
- **Gap: shared_abilities and detachment_abilities NOT counted** — flag for Phase 44 SYNC-02
- **Gap: rowCounts are client-side estimates, not DB actuals** — flag for Phase 44 SYNC-01
- Returns `{ wahapediaVersion, rowCounts }`
- `onSuccess` invalidates only 3 query keys: `RULES_SYNC_META_KEY`, `["datasheets-by-faction"]`, `["datasheet"]`
- **Gap: no invalidation for stratagems, detachments, detachment_abilities, shared abilities hooks** — flag for Phase 44 SYNC-05

Include a data flow diagram using ASCII art showing the 5 steps.

**Section 4: Migration Proposal (AUDIT-04)**

Per CONTEXT.md Decision #4: table name + column sketch + rationale. NOT full DDL.

**Group B — hobbyforge.db new tables (persist across re-syncs):**

```
Table: sync_errors (hobbyforge.db — NEW)
Purpose: Persistent sync error history with timestamp (META-04)
Phase: 45
Key columns:
  - id INTEGER PRIMARY KEY AUTOINCREMENT
  - occurred_at TEXT NOT NULL  — ISO 8601 timestamp
  - error_type TEXT NOT NULL   — e.g. "fetch_failed", "parse_error", "insert_error"
  - message TEXT NOT NULL      — human-readable error description
  - wahapedia_version TEXT     — version at time of error (may be null if fetch failed)
  - table_name TEXT            — which table the error occurred on (null for global errors)
Rationale: Must be in hobbyforge.db because rules.db is destroyed on re-sync. Error history survives app restarts and re-syncs.
```

```
Table: rules_snapshot (hobbyforge.db — NEW)
Purpose: Pre-sync snapshot for change detection (META-06, OVRD-06, OVRD-07)
Phase: 45
Key columns:
  - id INTEGER PRIMARY KEY AUTOINCREMENT
  - captured_at TEXT NOT NULL  — ISO 8601 timestamp
  - wahapedia_version TEXT     — version at time of capture
  - snapshot_type TEXT NOT NULL — "pre_sync" or "post_sync"
  - table_name TEXT NOT NULL   — which rw_* table this snapshot covers
  - row_count INTEGER          — number of rows in the table at capture time
  - checksum TEXT              — optional hash for quick diff detection
Design question: Full JSON serialization vs. per-table checksums — defer to Phase 45 research.
Rationale: Enables Phase 46 diff view by comparing pre-sync snapshot to post-sync state.
```
