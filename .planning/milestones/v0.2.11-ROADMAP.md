### Phase 70: Non-Destructive Recipe Save

**Goal**: Editing a recipe preserves all existing section and step database IDs; only genuinely changed fields are updated, only genuinely removed items are deleted
**Depends on**: Phase 69

**Plans**: 2 plans

Plans:
- [x] 70-01-PLAN.md — Add dbId to DraftStep/DraftSection, UpdateRecipeStepInput type, updateRecipeStep query, extend tests
- [x] 70-02-PLAN.md — Replace DELETE-all with three-way diff in RecipeFormSheet.onSubmit

## Milestone Summary

**Key Decisions:**
- better-sqlite3 (devDep) chosen over node:sqlite for tests — Vitest 4.x import-stripping bug (#7177)
- Non-destructive save requires dbId tracking in form state (five-phase diff algorithm)
- Direct assignment (not COALESCE) for nullable metadata updates
- ON DELETE SET NULL for session-section FK (dual-write with denormalized section_name)

**Issues Resolved:**
- COALESCE pattern prevented clearing nullable workflow metadata fields to null
- DELETE-all + re-INSERT recipe save destroyed section/step IDs on every edit
- Steps from different sections could interleave due to missing section-aware ordering
- Migrations 018-021 not registered in lib.rs — fresh install would fail
- Painting sessions had no durable FK to recipe sections

**Technical Debt Incurred:**
- UpdateRecipeStepInput uses Partial<> allowing omission of NOT NULL columns (single call site always passes all fields)
- 5 pre-existing test failures in unrelated rules-hub/datasheet tests
