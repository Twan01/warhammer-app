# Requirements: HobbyForge v2.0

**Defined:** 2026-05-01
**Core Value:** A single personal command center that always answers "what do I own, what's painted, and what's ready to play" — without ever depending on copyrighted GW data.

## v2.0 Requirements

Requirements for the Utility Layer milestone (Phases 6–9). Each maps to a roadmap phase.

### Foundation (Phase 6)

- [x] **STRAT-06**: Database migration 004 adds 8 nullable columns to `unit_strategy_notes` using `ALTER TABLE ... ADD COLUMN` statements only — no destructive operations, no edits to `001_core_schema.sql`, all pre-existing rows remain intact

### Paint Inventory (Phase 7)

- [x] **PINV-01**: User can navigate to a dedicated paint inventory page at `/paints` and see all paints in a filterable table with a color swatch (from `hex_color`) and a "used in N recipes" badge per row
- [x] **PINV-02**: User can filter paints by brand, paint type, and color family in any combination — filters reset when navigating away from the page
- [x] **PINV-03**: User can activate a "Running Low" preset view that shows only paints where `running_low = 1`; toggling off the preset restores all paints
- [x] **PINV-04**: User can activate a "Wishlist" preset view that shows only paints where `wishlist = 1`; both presets can be active simultaneously
- [x] **PINV-05**: User can click the "used in N recipes" badge and be navigated to the Recipes page pre-filtered to that paint via `/recipes?paintId=X`; zero-count badges are non-interactive
- [x] **PINV-06**: User can toggle a paint's `owned` status directly in the table row — the change takes effect immediately (optimistic update) and persists after navigation and app restart

### Army List Builder (Phase 8)

- [x] **ARMY-01**: User can create, rename, and delete army lists — each list has name (required), faction (optional), list type tag (Casual, Learning, Narrative, Competitive, Test), points limit, and notes
- [x] **ARMY-02**: User can add units from their collection to a list and remove them — each unit-in-list row shows painting status badge and assembled status (Assembled badge when `status_assembly = 1`)
- [x] **ARMY-03**: User can enter a per-unit points override; leaving it blank falls back to `unit.points`; effective points are computed via `COALESCE(points_override, unit.points, 0)` in SQL and are never reimplemented in JavaScript
- [x] **ARMY-04**: The army list detail view auto-calculates and displays: total points, painted points, and battle-ready percentage — updated immediately when units are added/removed or points overridden
- [x] **ARMY-05**: User can save per-list notes and per-unit-in-list notes without leaving the detail sheet — both field types save with toast confirmation
- [x] **ARMY-06**: The unit delete flow shows a warning with list membership count and list names before confirming when the unit belongs to one or more active army lists; units not in any list show a simple confirmation
- [x] **ARMY-07**: When no army lists exist, the page shows an empty state with a call-to-action button that opens the create sheet

### Unit Playbook (Phase 9)

- [x] **STRAT-01**: The unit detail sheet shows two tabs — "Details" (existing content) and "Playbook" (new) — with tab switching that works without closing and reopening the sheet; SheetFooter Edit/Delete buttons remain visible and functional on both tabs
- [x] **STRAT-02**: The Playbook tab shows a compact horizontal stats block with six integer fields (M, T, Sv, W, Ld, OC) — display mode shows values with appropriate suffixes (", +), edit mode shows raw number inputs; Escape cancels edit mode
- [x] **STRAT-03**: The Playbook tab has a multi-line Abilities text area (3 rows) and a single-line Keywords input — both accept free-text and save with the rest of the Playbook content
- [x] **STRAT-04**: The Playbook tab shows 8 strategy note text areas in fixed order: Battlefield Role, Optimal Range, Synergies, Counters, Deployment, Priority Targets, Mobility Notes, Personal Notes
- [x] **STRAT-05**: A Save button appears at the bottom of the Playbook tab and is disabled until content changes (dirty-state gating) — clicking Save persists all playbook fields in one `upsertStrategyNote` call, fires a success toast, and re-disables the button; errors fire an error toast
