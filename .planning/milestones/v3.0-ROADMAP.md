# HobbyForge v3.0 Roadmap — Personal Warhammer Command OS

## Purpose

HobbyForge has evolved beyond a simple collection tracker. The next major objective is to turn it into a **personal local-first Warhammer command operating system** that connects collection, painting, rules data, points, army lists, game preparation and post-game learning.

The core product promise is:

> Move from “I own models” to “I am ready to play, I know what to paint next, I understand my list, and I learn from every game.”

This roadmap is intentionally **additive**. It must extend existing systems rather than rebuild them.

---

## Non-negotiable constraints

1. The app remains personal and local-first.
2. External imported data is treated as user-provided local data.
3. The app source must not bundle copyrighted Games Workshop rules, points, codex text or artwork.
4. Imported rules/points data must be traceable, versioned and user-verifiable.
5. Manual overrides must always be possible.
6. Never silently present imported data as guaranteed current/official.
7. Preserve the existing Tauri + React + SQLite architecture.
8. Preserve sibling portal patterns for sheets/dialogs.
9. Prefer additive migrations.
10. Do not duplicate existing systems.

# Milestone 3.1 — Recipes 2.0 / Painting Studio

## Goal

Transform recipes from simple paint notes into a structured painting knowledge system connected to units, projects, paint inventory, photos and sessions.

## Why this matters

The user paints and collects before playing. A strong recipe system creates immediate value and supports future list-to-paint planning.

## Requirements

### Recipe metadata

Extend recipes to support:

- Style: clean, grimdark, speedpaint, display, tabletop, tabletop+, eavy-metal inspired, custom.
- Surface: armor, skin, cloth, weapon, leather, metal, lens, glow, base, weathering, other.
- Effect: plasma glow, rust, dust, battle damage, NMM, OSL, edge highlight, recess shade, other.
- Difficulty: easy, medium, advanced.
- Estimated total minutes.
- Result photo.
- Linked faction.
- Linked unit.
- Linked active project where relevant.
- Source/tutorial link.
- Free-form notes.

### Recipe steps

Create or extend a step model with:

- Step order.
- Step title.
- Phase: prime, basecoat, shade, layer, highlight, glaze, weathering, basing, varnish, other.
- Paint used.
- Optional alternative paint.
- Tool: brush, drybrush, airbrush, sponge, oil/enamel, pigment, other.
- Dilution.
- Technique.
- Notes.
- Estimated minutes.
- Step photo.

### Paint inventory integration

For each recipe, show:

- Required paints.
- Owned paints.
- Missing paints.
- Running-low paints.
- Paint substitutions.
- Add missing paints to wishlist.

### UX upgrades

Upgrade Recipes from a table-first experience into a studio experience:

- Recipe cards with color swatches.
- Detail view as step-by-step timeline.
- Filters by faction, unit, surface, style, difficulty and missing paints.
- Duplicate recipe action.
- Start from template action.
- Used-in projects section.

### Integration

Surface relevant recipes in:

- Current Focus card.
- Unit detail sheet.
- Kanban cards.
- Log Session sheet.

When logging a painting session, allow the user to select:

- Recipe used.
- Step worked on.
- Paints used.
- Step completion.

## Acceptance criteria

- Existing recipe data is preserved.
- Existing `recipe_paints` links are preserved or migrated.
- User can create a structured recipe with steps.
- User can see owned/missing paints.
- User can link a recipe to faction/unit/project.
- Relevant recipes appear in Current Focus and Kanban.
- No existing recipe functionality regresses.

# Milestone 3.2 — Rules Sync 2.0 / Rules Data Hub

## Goal

Stabilize and extend the existing local rules import architecture so HobbyForge becomes a reliable personal rules and points reference.

## Requirements

Before implementation:

- Confirm current sync path.
- Confirm existing `rules.db` schema.
- Confirm whether wargear tables exist and are populated.
- Confirm whether stratagems, detachments and detachment abilities are migrated and synced.

Add or complete support for:

- Datasheet wargear.
- Shared abilities.
- Stratagems.
- Detachments.
- Detachment abilities.
- Points data if a reliable source/format is available.
- Import logs.
- Import source registry.

Track:

- Source name.
- Source URL.
- Sync date.
- Source version.
- Row counts.
- Last successful sync.
- Error/failure logs.
- Rules edition.
- Warnings.

### Manual overrides

Support visible manual overrides for:

- Points.
- Unit stats.
- Keywords.
- Ability reminders.
- Loadout points.
- List-specific points.

Support future comparison for:

- Points changed.
- Unit stats changed.

# Milestone 3.3 — Points and Army Lists 2.0

## Goal

Upgrade Army Lists from points-tracked lists into true game-preparation objects.

## Current baseline

Army Lists already exist and already support effective points, points overrides, list units and battle-ready summaries.

## Requirements

### Ruleset-aware lists

Extend lists with:

- Ruleset/source version.
- Points source.
- Detachment.
- Game size: Combat Patrol, Incursion, Strike Force, Onslaught, custom.
- List status: draft, planned, ready, played, archived.

### Advanced points handling

Support:

- Imported points.
- Unit point tiers.
- Loadout-specific points.
- Manual points override.
- Warning when points are outdated.
- Warning when unit has no points.
- Points delta after sync.

### Owned vs wishlist

Allow list entries to be:

- Owned unit from collection.
- Wishlist / planned unit.
- Proxy placeholder.
- Duplicate squad.

Show:

- Owned points.
- Missing points.
- Painted points.
- Unpainted points.
- Battle-ready points.
- Wishlist cost estimate where known.

### List readiness

For each list, show:

- Ownership readiness.
- Assembly readiness.
- Painting readiness.
- Battle-ready readiness.
- Rules-data freshness.

### Tactical tags

Add manual tactical tags for units/list entries:

- Anti-tank.
- Anti-infantry.
- Anti-elite.
- Objective holder.
- Screening.
- Fast mover.
- Deep strike.
- Durable.
- Buff piece.
- Melee threat.
- Shooting threat.
- Transport.
- Indirect.
- Action unit.
- Trading piece.

### List analysis

Show soft warnings:

- Low anti-tank.
- Low objective control.
- Few fast units.
- Too many unpainted points.
- Too many missing units.
- Outdated points.
- No rules notes.
- Missing playbook reminders.

### Export / print

Create a printable list view with:

- List name.
- Faction.
- Detachment.
- Total points.
- Units.
- Points.
- Status/readiness.
- Key reminders.
- User notes.

## Acceptance criteria

- Existing army lists migrate cleanly.
- User can create lists from owned and wishlist units.
- Points are calculated correctly from imported, tiered, loadout or override data.
- Readiness is visible at list level.
- Tactical gaps are visible.
- Export/print view is available or scaffolded.

# Milestone 3.5 — Battle Intelligence 2.0

## Goal

Turn Battle Log into a learning and improvement system.

## Current baseline

Battle Log already records result, opponent, mission, score, MVP, underperformer, lessons and changes next time.

## Requirements

### Battle report upgrade

Add optional fields:

- Deployment type.
- First turn.
- Primary score.
- Secondary score.
- Battle size.
- Opponent archetype.
- Key turning point.
- Biggest mistake.
- Best decision.
- Rule forgotten.
- Photo.
- Confidence/emotional rating.

### Unit performance

For each unit in the list, allow after-game evaluation:

- Performance rating.
- Role fulfilled?
- Damage output perception.
- Survivability.
- Objective contribution.
- Notes.
- Keep/change/drop.

### Analytics

Show:

- Win/loss/draw by faction.
- Win rate by own list.
- Win rate by opponent faction.
- Average score.
- Common MVP units.
- Common underperformers.
- Recurring lessons.
- Recurring mistakes.
- Best performing list archetypes.

### Feedback loop

Battle logs should feed:

- Unit playbook.
- List analysis.
- Game Day reminders.
- Future list suggestions.

Example: if the user repeatedly logs “forgot stratagem X”, Game Day Mode should surface it as a reminder.

## Acceptance criteria

- Existing battle logs remain valid.
- User can record richer battle reports.
- Unit performance is tracked over time.
- Battle analytics are visible.
- Lessons can feed back into Playbook/Game Day reminders.

# Milestone 3.6 — Smart Planning / Next Best Action

## Goal

Make HobbyForge actively guide the user.

## Requirements

Create a rule-based recommendation engine combining:

- Active projects.
- Next planned game.
- Army list readiness.
- Missing units.
- Painting progress.
- Recipes.
- Paint inventory.
- Wishlist.
- Spending.
- Rules data freshness.

Example recommendations:

```text
Recommended next actions:
1. Paint Tau Fire Warriors to complete your 500pt list.
2. Buy White Scar — needed for 3 active recipes.
3. Update points for Crisis Battlesuits — last verified 3 months ago.
4. Log rules notes for your Commander before your first game.
```

### Prioritization logic

Prioritize actions that:

- Unlock a playable list.
- Prepare an upcoming game.
- Reduce unpainted backlog.
- Complete a current project.
- Fix outdated rules/points.
- Fill a tactical list gap.

## Acceptance criteria

- Dashboard can show top 3 recommended actions.
- Recommendations are explainable.
- User can click into the relevant object.
- Initial version can be rule-based; no AI required.

# Product principle

Do not add features because they are interesting. Every feature must support at least one of these goals:

1. Help the user prepare a real game.
2. Help the user know what to paint next.
3. Help the user understand their rules/list.
4. Help the user avoid forgetting important rules.
5. Help the user learn from games played.
6. Help the user manage collection, backlog and spending.

---

# Final target

HobbyForge v3.0 should become:

> A personal local-first Warhammer command OS combining collection, painting, rules, points, list preparation, game-day reminders and post-game learning.

It should not be just a database. It should become a practical assistant for moving from “I own models” to “I am ready to play and improve.”
