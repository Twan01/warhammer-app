**Title:** `# Points Import Design`
**Subtitle:** `Design document for PTS-01 through PTS-04 (implementation deferred to future milestone)`

**Required sections (all 5 must be present):**

### 1. Schema Design (`## Schema`)
Document a new `points_imports` table in hobbyforge.db:
- `id INTEGER PRIMARY KEY AUTOINCREMENT`
- `unit_name TEXT NOT NULL` — matches unit name for linking (same TEXT copy pattern as unit_overrides)
- `faction_id TEXT` — Wahapedia faction_id for scoping
- `points INTEGER NOT NULL` — imported points value
- `source TEXT NOT NULL DEFAULT 'csv'` — origin of the data
- `imported_at TEXT NOT NULL DEFAULT (datetime('now'))` — when this import occurred
- `version TEXT` — optional version tag for the import batch
- `UNIQUE (unit_name, faction_id)` — one points entry per unit per faction

Document a `points_import_history` table for tracking import batches:
- `id INTEGER PRIMARY KEY AUTOINCREMENT`
- `imported_at TEXT NOT NULL DEFAULT (datetime('now'))`
- `source_file TEXT` — original filename
- `version TEXT` — user-provided version label
- `row_count INTEGER NOT NULL DEFAULT 0`
- `delta_added INTEGER NOT NULL DEFAULT 0`
- `delta_removed INTEGER NOT NULL DEFAULT 0`
- `delta_changed INTEGER NOT NULL DEFAULT 0`

Both tables live in hobbyforge.db (not rules.db) — they persist across rules syncs.

### 2. Versioning (`## Versioning`)
Document: Each import batch gets a version label (user-provided or auto-generated from date). The `points_import_history` table tracks every import with row counts and deltas. Previous points are overwritten (INSERT OR REPLACE on UNIQUE constraint), not version-stacked. The history table provides audit trail of what changed when.

### 3. Delta Computation (`## Delta Computation`)
Document the algorithm:
1. Before import, snapshot current `points_imports` rows for the affected faction(s)
2. Process CSV rows, INSERT OR REPLACE into `points_imports`
3. Compare before/after: count added (new unit_name), removed (unit_name no longer present), changed (same unit_name but different points value)
4. Store delta counts in `points_import_history`
5. For PTS-02 UI: query `points_import_history` for the most recent import to show delta summary

### 4. Manual Override Interaction (`## Manual Override Interaction`)
Document the COALESCE precedence chain:
```
effective_points = COALESCE(alu.points_override, pi.points, uo.points, u.points, 0)
```
Where `pi.points` is from `points_imports` (new), `uo.points` is from `unit_overrides` (existing), and `u.points` is from `units` table (original).

Key rule: `alu.points_override` (user's per-list manual override) ALWAYS wins. `points_imports` sits between manual override and unit_overrides in the precedence chain.

Document: When a user has set points_override on a specific army list unit, imported points changes do NOT affect that unit's effective_points. The override is explicit user intent and takes absolute priority (PTS-04).

### 5. Army List Impact (`## Army List Impact`)
Document:
- After a points import, army lists that contain affected units will show updated totals (via the COALESCE chain) on next query
- For PTS-03: Compare `points_imports.imported_at` against a staleness threshold (e.g., 30 days). If the most recent import is older than threshold, show a warning badge on affected army lists
- Units with manual overrides (points_override IS NOT NULL) are unaffected by imports — their effective_points remain locked to the override value
- The `getArmyListWithUnits` query in armyLists.ts would gain a LEFT JOIN to `points_imports` to include imported points in the COALESCE chain

Add a final section `## Implementation Notes`:
- This design is for documentation only — implementation is deferred to a future milestone
- No migration is created by this document
- The COALESCE chain modification in armyLists.ts would require updating the JOIN in `getArmyListWithUnits` and `getArmyListReadiness`
- CSV parsing can reuse the existing `src/lib/csv.ts` utilities from Wahapedia sync
