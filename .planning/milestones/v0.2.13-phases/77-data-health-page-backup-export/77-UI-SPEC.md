## Typography

| Role | Size | Weight | Line Height | Differentiation |
|------|------|--------|-------------|-----------------|
| Caption | 12px (`text-xs`) | 400 (regular) | 1.5 | Used for metadata labels (`text-muted-foreground uppercase tracking-wider`) |
| Body | 14px (`text-sm`) | 400 (regular) | 1.5 | Default text, descriptions, diagnostic messages |
| Heading | 20px (`text-xl`) | 600 (semibold) | 1.2 | Section titles |
| Display | 28px (`text-3xl`) | 600 (semibold) | 1.2 | Large metric values in StatCards |

**Focal point:** The 28px display numbers on StatCards in the Table Counts grid are the primary focal point of the page, drawing the eye to row counts as the key data health indicator.

**Weight constraint:** Only 2 weights used: 400 (regular) and 600 (semibold). Labels and captions are differentiated from body text via size (12px vs 14px), color (`text-muted-foreground`), and letter-spacing (`uppercase tracking-wider`) rather than a separate font weight.

Display (28px) is used for the large metric values in row-count cards (matching existing `StatCard` `text-3xl font-semibold`). Heading (20px) is used for section titles ("Version Info", "Table Counts", "Diagnostics", "Backup"). Body (14px) is used for descriptions and diagnostic messages. Caption (12px) is used for metadata labels in the Version & Schema Info section.

### Semantic Status Colors

This page uses semantic status colors for diagnostic severity and freshness indicators. These follow established patterns from `SyncStatusCard` and `syncFreshness.ts`:

| Status | Tailwind Class | Usage |
|--------|---------------|-------|
| Healthy / Fresh | `text-green-500` / `bg-green-500` | Freshness dot when sync < 7 days, zero-issue diagnostics |
| Warning / Stale | `text-amber-500` / `bg-amber-500` | Freshness dot when sync 7-30 days, diagnostic warnings |
| Error / Critical | `text-red-500` / `bg-red-500` | Freshness dot when sync > 30 days, orphaned rows |
| Info | `text-blue-500` / `bg-blue-500` | Informational diagnostic flags (row count anomalies) |
| Muted | `text-muted-foreground` | Secondary descriptions, timestamps |

Accent reserved for: Nothing on this page. The Data Health page has no faction context so `--faction-accent` is not applied. All status indication uses the semantic status colors above.

## Page Layout

### Overall Structure

Single-column scrollable page. No sidebar detail panel. Four stacked sections in this order:

1. **Page Header** -- Page title "Data Health" with version summary inline
2. **Version & Schema Info** -- Compact horizontal row of metadata
3. **Table Counts** -- Grid of metric cards (reuse `StatCard` pattern)
4. **Diagnostics** -- List of diagnostic flag cards with severity badges
5. **Backup** -- Backup action card with status

### Section 1: Page Header

- Page title: `h1` at 20px semibold, matching other page headers in the app
- Inline version string below title: `text-sm text-muted-foreground`
- Format: "HobbyForge v{version} -- Schema v{user_version} -- Rules v{rules_user_version}"
- Version data loads via `getVersion()` and `PRAGMA user_version`; show `Skeleton` (w-48 h-4) while loading

### Section 2: Version & Schema Info

A single `Card` containing a horizontal flex row of key-value pairs:

| Label | Value Source | Format |
|-------|-------------|--------|
| App Version | `getVersion()` | `v0.2.13` |
| DB Schema | `PRAGMA user_version` on hobbyforge.db | `v28` |
| Rules Schema | `PRAGMA user_version` on rules.db | `v4` |
| Last Sync | `rw_sync_meta.last_sync_at` | Relative date via `getSyncAgeLabel()` + freshness dot |
| Sync Errors | `useRulesSyncErrors` count | Badge with count, `variant="destructive"` if > 0, `variant="secondary"` if 0 |

Layout: Horizontal flex with `gap-8`, items wrap on narrow windows. Each item is a vertical stack: label (`text-xs text-muted-foreground uppercase tracking-wider`) over value (`text-sm font-semibold`).

Loading state: Each value slot shows `Skeleton` (w-16 h-4) independently.

### Section 3: Table Counts

Grid of 5 metric cards using the existing `StatCard` component pattern (28px semibold number, 14px muted label below).

Grid: `grid grid-cols-2 md:grid-cols-3 lg:grid-cols-5 gap-4`

| Card | Label | Query |
|------|-------|-------|
| 1 | "Units" | `SELECT COUNT(*) FROM units` |
| 2 | "Recipes" | `SELECT COUNT(*) FROM painting_recipes` |
| 3 | "Assignments" | `SELECT COUNT(*) FROM unit_recipe_assignments` |
| 4 | "Step Progress" | `SELECT COUNT(*) FROM unit_recipe_step_progress` |
| 5 | "Synced Points" | `SELECT COUNT(*) FROM synced_unit_points` |

Loading state: Each card shows `Skeleton` (w-12 h-8) for the number and `Skeleton` (w-20 h-4) for the label.

No `animate` prop on StatCards (these are not dashboard hero stats). No `to` prop (not navigable).

### Section 4: Diagnostics

A `Card` containing a vertical list of diagnostic results. Each diagnostic is a row:

```
[severity badge]  [description text]                    [count]
```

- Severity badge: `Badge` component
  - `variant="destructive"` for warnings (orphaned rows, ambiguous matches)
  - `variant="secondary"` for info (stale sync)
- Description: 14px regular, `text-foreground`
- Count: 14px semibold tabular-nums, right-aligned
- Each row separated by `border-b border-border/40` (last row no border)
- Row padding: `py-2` (8px from declared spacing scale)

Diagnostic items:

| ID | Description (healthy) | Description (flagged) | Severity |
|----|----------------------|----------------------|----------|
| orphaned-progress | "All progress rows have matching steps" | "{n} orphaned progress rows -- tracking completion for steps that no longer exist" | warning |
| ambiguous-points | "All units have unambiguous point matches" | "{n} units have ambiguous or missing point matches" | warning |
| stale-sync | "Rules data is up to date" | "Rules data is {age} old -- consider re-syncing" | info |

When all diagnostics are healthy (zero issues), show a single consolidated row:
- Green circle dot (8px, `bg-green-500`) + "All diagnostics passed" in `text-sm text-muted-foreground`

Loading state: Three rows of `Skeleton` (full-width h-10) with `gap-2`.

Empty state: Not applicable (diagnostics always return results even if all healthy).

### Section 5: Backup

A `Card` with:

**Top row (flex between):**
- Left: "Database Backup" heading (14px semibold) + last backup status below
- Right: "Create Backup" button (primary variant, `Download` icon from Lucide)

**Last backup status** (below heading):
- If backup exists in localStorage: `text-sm text-muted-foreground` -- "Last backup: {date} -- {filename}"
- If no backup recorded: `text-sm text-muted-foreground` -- "No backups yet"

**During backup (button state):**
- Button disabled, text changes to "Creating backup...", icon shows `Loader2` with `animate-spin`

**After successful backup:**
- Toast notification via `sonner`: "Backup created successfully"
- Last backup status updates immediately from localStorage

## Interaction Contract

### Backup Flow

1. User clicks "Create Backup"
2. System opens native save dialog (via `tauri-plugin-dialog`) with default filename `hobbyforge-backup-YYYY-MM-DD.db`
3. If user cancels dialog: no action, no toast, no state change
4. If user confirms: button enters loading state (disabled, spinner icon, "Creating backup..." text)
5. Backend runs `VACUUM INTO` to the chosen path
6. On success: toast "Backup created successfully", update localStorage with `{ date, path, filename }`, re-render BackupCard status
7. On error: toast "Backup failed: {message}", button returns to idle state

### Async Loading

All three data sections (version info, table counts, diagnostics) load independently via separate React Query hooks. The page renders immediately with skeleton placeholders. Each section resolves on its own timeline. There is no global loading gate.

### No Destructive Actions

This phase contains zero destructive actions. No delete, no "fix" buttons, no data modification. The page is read-only except for the backup creation (which is additive, writing a new file).
