<objective>
Wire the Log Session sheet to the applied recipe system so that logging a painting session against a recipe step automatically marks that step as completed in the unit's applied recipe assignment.

Purpose: Without this bridge, users must manually check off steps in the applied recipe checklist even after logging a session against that exact step -- a frustrating double-entry UX gap.
Output: Modified LogSessionSheet.tsx with bridge logic in onSubmit.
</objective>

    Implement the step auto-completion bridge in LogSessionSheet per D-01 through D-04.

    **New imports:** Add useAssignmentsByUnit, useCreateAssignment, useToggleStepProgress, and ASSIGNMENTS_KEY from "@/hooks/useRecipeAssignments". Also import useQueryClient from "@tanstack/react-query" (if not already imported).

    **New hooks at component level** (add alongside the existing watchedRecipeId block near line 131):
    - watchedUnitId = form.watch("unit_id")
    - const { data: unitAssignments = [] } = useAssignmentsByUnit(watchedUnitId > 0 ? watchedUnitId : undefined)
    - const createAssignment = useCreateAssignment()
    - const toggleStepProgress = useToggleStepProgress()
    - const qc = useQueryClient()

    Guard watchedUnitId with > 0 check to prevent query when no unit is selected (matches useAssignmentsByUnit enabled: unitId !== undefined contract).

    **Bridge block in onSubmit:** Insert AFTER createSession.mutateAsync succeeds (after line ~181) and BEFORE the existing if (values.new_status) block (line ~182). The bridge runs only when values.recipe_id != null AND values.recipe_step_id != null.

    Bridge logic:
    1. Find the step in recipeSteps (already in scope) by id === values.recipe_step_id to get step.order_index. CRITICAL: Use step.order_index as the orderIndex argument, NOT values.recipe_step_id (Pitfall 2 in RESEARCH.md).
    2. Check unitAssignments.find(a => a.recipe_id === values.recipe_id) for an existing assignment.
    3. If no assignment found (D-02): call createAssignment.mutateAsync({ unit_id: values.unit_id, recipe_id: values.recipe_id }) to get the new assignment id.
    4. Call toggleStepProgress.mutateAsync({ assignmentId: assignmentId, orderIndex: step.order_index, completed: true }) (D-04: only the specific step).
    5. After success: call qc.invalidateQueries({ queryKey: ASSIGNMENTS_KEY }) for broad invalidation so Kanban and CurrentFocusCard pick up the new progress. Also invalidate queryKey ["kanban-enrichment"] prefix.
    6. Wrap entire bridge in try/catch. On catch: toast.warning("Session logged but step progress update failed."), call onClose(), return. This is the D-03 partial-failure pattern matching the existing status update block.

    **Do NOT modify** the existing status update block (values.new_status), the existing createSession.mutateAsync call, or the success toast logic. The bridge sits between session creation and status update as a new independent side-effect.
