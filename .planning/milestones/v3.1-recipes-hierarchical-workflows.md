# HobbyForge v3.1 — Recipes 2.0: Hierarchical Painting Workflows

## Purpose

The current recipe implementation is a useful first step: recipes can now have ordered steps, linked paints, drag-and-drop ordering and missing/owned paint indicators.

However, the current model is still too flat:

```text
Recipe
  Step
  Step
  Step
```

This does not fully reflect real miniature painting workflows. Hobbyists do not always paint every basecoat first, then every shade, then every layer. Sometimes the correct process is to complete one surface, subassembly or technique block first.

Example: Ultramarines armor with sponge weathering may require completing the entire blue armor workflow before moving to trim, weapons, lenses or basing.

The target model is:

```text
Recipe
  Workflow / Section
    Step
    Step
    Step
  Workflow / Section
    Step
    Step
    Step
```

This allows HobbyForge to represent realistic painting sequences such as:

- Armor Blue workflow.
- Gold Trim workflow.
- Weapon Casing workflow.
- Lenses workflow.
- Leather workflow.
- Base workflow.
- Sponge Weathering workflow.
- Airbrush Stage workflow.
- Decal + Weathering workflow.
- Character Upgrade workflow.

---

## Current baseline

The current implementation includes:

- `painting_recipes` table.
- `recipe_paints` table acting as the current step/link table.
- `RecipeFormSheet` with draft steps.
- `RecipeStepList` with drag-and-drop using `@dnd-kit`.
- `RecipeStepRow` with step name, paint combobox and notes.
- Inline paint creation from recipe step row.
- `RecipeDetailSheet` displaying ordered steps and owned/missing paint indicators.
- `RecipesPage` with table-first browsing and filters by faction, unit, area and paint.

Known limitation:

- A recipe is currently a single ordered list of steps.
- Steps are not grouped by surface, workflow, technique or subassembly.
- Step metadata is minimal: `step_name`, `paint_id`, `order_index`, `notes`.
- The current model cannot properly express “complete the whole armor workflow before moving to other areas”.

---

## Product goal

Turn recipes into a flexible **Painting Studio workflow system** that supports both simple and advanced painting methods.

The system must allow the user to model real workflows, including:

- Completing one surface before another.
- Completing one technique block before another.
- Painting subassemblies separately.
- Batch painting all models step-by-step.
- Completing one model fully before the next.
- Adding optional display-level upgrade steps.
- Tracking recipe progress through Log Session and Kanban later.

---

## Non-negotiable constraints

1. Do not destroy existing recipe data.
2. Existing `recipe_paints` data must migrate safely.
3. Keep the implementation additive first.
4. Do not overcomplicate the UI.
5. Keep existing recipe flows working during transition.
6. Preserve TypeScript strict mode.
7. Preserve current sibling portal architecture.
8. Add tests for migration and backward compatibility.
9. Prefer incremental migration from flat steps to hierarchical workflows.

## Target concept

### Level 1 — Recipe

A full paint scheme or recipe.

Examples:

- Ultramarines — Grimdark Blue Armor & Urban Base.
- Tau — Clean White Armor and Blue Lenses.
- Necron — Aged Metal and Green Glow.

Recipe-level metadata should eventually support:

- name;
- faction;
- unit;
- style;
- difficulty;
- intended quality;
- estimated total time;
- tutorial/source link;
- notes;
- result photo.

### Level 2 — Workflow / Section

A coherent surface, technique block, subassembly or effect.

Examples:

- Armor Blue.
- Gold Trim.
- Weapon Casing.
- Lenses.
- Base.
- Sponge Weathering.
- Airbrush Stage.
- Decals.
- Final Weathering.

A workflow/section should support:

- `id`;
- `recipe_id`;
- `name`;
- `section_type`;
- `surface`;
- `technique`;
- `execution_mode`;
- `applies_to`;
- `order_index`;
- `estimated_minutes`;
- `optional`;
- `notes`;
- `created_at`;
- `updated_at`.

Recommended enum-style values:

```text
section_type:
  surface | technique | subassembly | basing | effect | global | other

surface:
  armor | skin | cloth | weapon | leather | metal | lens | glow | base | weathering | other

technique:
  brush | drybrush | airbrush | sponge | oil_enamel | pigment | decal | mixed | other

execution_mode:
  complete_section_first | batch_step_across_models | model_by_model | optional_upgrade | conditional

applies_to:
  all_models | selected_models | character_only | vehicle_only | custom
```

### Level 3 — Step

An individual painting action within a workflow.

A step should support:

- `id`;
- `recipe_id`;
- `section_id`;
- `title`;
- `phase`;
- `primary_paint_id`;
- `alternative_paint_id`;
- `tool`;
- `dilution`;
- `technique`;
- `target_area`;
- `order_index`;
- `estimated_minutes`;
- `photo_asset_id`;
- `optional`;
- `depends_on_step_id`;
- `notes`;
- `created_at`;
- `updated_at`.

Recommended enum-style values:

```text
phase:
  prime | basecoat | shade | layer | highlight | glaze | weathering | basing | varnish | other

tool:
  brush | drybrush | airbrush | sponge | oil_enamel | pigment | decal | other
```

## Proposed database design

### Additive migration

Create a new migration, for example:

```text
src-tauri/migrations/012_recipe_workflows.sql
```

Recommended tables:

```sql
CREATE TABLE IF NOT EXISTS recipe_sections (
    id                INTEGER PRIMARY KEY AUTOINCREMENT,
    recipe_id          INTEGER NOT NULL REFERENCES painting_recipes(id) ON DELETE CASCADE,
    name              TEXT NOT NULL,
    section_type      TEXT,
    surface           TEXT,
    technique         TEXT,
    execution_mode    TEXT,
    applies_to        TEXT,
    order_index       INTEGER NOT NULL DEFAULT 0,
    estimated_minutes INTEGER,
    optional          INTEGER NOT NULL DEFAULT 0,
    notes             TEXT,
    created_at        TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at        TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS recipe_steps (
    id                   INTEGER PRIMARY KEY AUTOINCREMENT,
    recipe_id             INTEGER NOT NULL REFERENCES painting_recipes(id) ON DELETE CASCADE,
    section_id            INTEGER REFERENCES recipe_sections(id) ON DELETE CASCADE,
    title                 TEXT NOT NULL,
    phase                 TEXT,
    primary_paint_id      INTEGER REFERENCES paints(id) ON DELETE RESTRICT,
    alternative_paint_id  INTEGER REFERENCES paints(id) ON DELETE RESTRICT,
    tool                  TEXT,
    dilution              TEXT,
    technique             TEXT,
    target_area           TEXT,
    order_index           INTEGER NOT NULL DEFAULT 0,
    estimated_minutes     INTEGER,
    photo_asset_id        INTEGER REFERENCES image_assets(id) ON DELETE SET NULL,
    optional              INTEGER NOT NULL DEFAULT 0,
    depends_on_step_id    INTEGER REFERENCES recipe_steps(id) ON DELETE SET NULL,
    notes                 TEXT,
    created_at            TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at            TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### Migration from existing data

For each existing `painting_recipes` row:

1. Create one default `recipe_sections` row.
2. Section name should be:
   - `recipe.area` if present;
   - otherwise `General workflow`.
3. For each existing `recipe_paints` row:
   - create one `recipe_steps` row;
   - map `recipe_paints.step_name` to `recipe_steps.title`;
   - map `recipe_paints.paint_id` to `recipe_steps.primary_paint_id`;
   - map `recipe_paints.order_index` to `recipe_steps.order_index`;
   - map `recipe_paints.notes` to `recipe_steps.notes`;
   - set `section_id` to the default section.
4. Do not delete `recipe_paints` yet.

### Backward compatibility

The first implementation should keep existing `recipe_paints` untouched until the new model is verified.

Possible approach:

- New UI reads from `recipe_sections` / `recipe_steps` when available.
- If no sections exist for a recipe, fall back to old `recipe_paints` and/or auto-create default section.
- Once stable, deprecate old `recipe_paints` usage in application code.

### Recipe form UX

The form should not become overwhelming.

Recommended UX:

- Recipe-level fields remain at the top.
- Workflows/sections are collapsible cards.
- Each section shows summary metadata compactly.
- Advanced section metadata can be hidden under “Advanced”.
- Steps inside a section remain easy to add/reorder.
- Step advanced fields can also be collapsed.

### Section card design

Each section card should show:

- section name;
- section type;
- surface;
- technique;
- execution mode;
- estimated time;
- step count;
- missing paint count;
- optional badge if applicable.

  4. Recess shade

Gold Trim

  2. Wash brown

Show:

- paint swatches;
- owned/missing/running-low state;
- estimated time;

## Acceptance criteria

The implementation is successful if:

1. Recipes support multiple workflow sections.
2. Each section can contain ordered steps.
3. Existing recipe data is preserved and migrated.
4. The user can model workflows like “complete all blue armor first, then gold trim, then lenses”.
5. The UI remains usable and not overwhelming.
6. Missing/owned paint indicators still work.
7. Recipe detail displays a clear workflow timeline.
8. Drag-and-drop ordering still works at least within sections.
9. TypeScript build passes.
10. Tests cover migration, queries and core UI behavior.

## Out of scope for the first implementation

These can come later:

- Full Log Session step completion tracking.
- Per-step completion history.
- Progress by recipe section.
- Moving steps between sections via drag-and-drop.
- Photo upload per step if image integration becomes too large.
- Advanced dependency graph between steps.
- AI-assisted recipe generation.

---

## Final product principle

The system should not force hobbyists into a global basecoat/layer/highlight sequence.

It should support real painting behaviour:

> Sometimes the right workflow is to finish one surface, one technique block, or one subassembly completely before moving to the next part of the miniature.

HobbyForge recipes should become flexible painting workflows, not just paint checklists.
