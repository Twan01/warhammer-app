# Milestone v0.2.9: Recipes 3.1 / Workflow Semantics & Integrations

**Status:** SHIPPED 2026-05-12
**Phases:** 57-60
**Total Plans:** 8

## Overview

Make recipe sections semantically rich and actionable — extending workflow metadata into Log Session, Kanban, and Current Focus for real painting workflow guidance.

## Phases

### Phase 57: Schema & Data Layer

**Goal**: Recipe sections carry workflow semantics and painting sessions can reference which section was worked on
**Depends on**: Phase 56 (v0.2.8 complete)
**Requirements**: WF-01, WF-02, WF-03, WF-04, WF-05
**Plans**: 2 plans

Plans:
- [x] 57-01-PLAN.md — Migration + TypeScript types + const arrays
- [x] 57-02-PLAN.md — Query functions + DraftSection + save path + tests

**Details:**
- Migration 020: 4 TEXT DEFAULT NULL columns on recipe_sections (section_type, technique, execution_mode, applies_to) + section_name on painting_sessions
- Const arrays: SECTION_TYPES (7), TECHNIQUES (9), EXECUTION_MODES (3) with derived type aliases
- Full round-trip: DraftSection → createRecipeSection → RecipeSection with null-coalesce guards
- Denormalized section_name on PaintingSession (no FK — DELETE-all + re-INSERT pattern)

### Phase 58: Recipe Form & Timeline Display

**Goal**: Users can edit workflow metadata on recipe sections and see it at a glance in the timeline view
**Depends on**: Phase 57
**Requirements**: RUI-01, RUI-02, RUI-03, RUI-04
**Plans**: 2 plans

Plans:
- [x] 58-01-PLAN.md — Workflow collapsible form controls in RecipeSectionCard (RUI-01, RUI-02)
- [x] 58-02-PLAN.md — Timeline workflow metadata display in SectionedTimeline (RUI-03, RUI-04)

**Details:**
- RecipeSectionCard: progressive disclosure "Workflow" collapsible with 4 Select controls
- Simple recipes (single section, no metadata) hide collapsible entirely
- SectionedTimeline: section_type as Badge, execution_mode as dot-separated text (design deviation D-08)
- Technique inline display in dot-separated metadata string

### Phase 59: Session Section Cascade

**Goal**: Users can log painting sessions with section-level granularity through a natural cascading selector flow
**Depends on**: Phase 57
**Requirements**: SESS-01, SESS-02, SESS-03, SESS-04, SESS-05
**Plans**: 2 plans

Plans:
- [x] 59-01-PLAN.md — Schema extension: section_name field + SESS-05 schema tests
- [x] 59-02-PLAN.md — LogSessionSheet cascade selector + component tests (SESS-01 through SESS-05)

**Details:**
- 3-level cascade: recipe → section → step with useState local filter state
- Reset chain 1 (recipe change): clears section_name + watchedSectionId + recipe_step_id
- Reset chain 2 (section change): clears recipe_step_id only
- Section selector conditionally rendered when recipe selected AND sections.length >= 2
- filteredSteps useMemo filters by watchedSectionId when non-null

### Phase 60: Kanban & CurrentFocus Integration

**Goal**: Users see section-aware workflow context on project cards and dashboard, knowing exactly where they are in a recipe
**Depends on**: Phase 57, Phase 58, Phase 59
**Requirements**: PROJ-01, PROJ-02, PROJ-03, PROJ-04, PROJ-05
**Plans**: 2 plans

Plans:
- [x] 60-01-PLAN.md — Pure function computeWorkflowPosition + batch hook useWorkflowPositions + tests (PROJ-03, PROJ-04, PROJ-05)
- [x] 60-02-PLAN.md — KanbanCard + CurrentFocusCard workflow display integration + component tests (PROJ-01, PROJ-02)

**Details:**
- computeWorkflowPosition: pure function handling sectioned/flat/section-only/complete/orphaned scenarios
- useWorkflowPositions: batch React Query hook (sorted ID key, Map result, 5min staleTime)
- KanbanCard: 5 rendering paths (section+step, section-only, flat step N/M, complete, fallback to status hint)
- CurrentFocusCard: 5 rendering paths with technique support and Layers icon prefix
- Cache invalidation: useCreatePaintingSession invalidates workflow-positions key

## Milestone Summary

**Key Decisions:**
- DELETE-all + re-INSERT save pattern on recipe sections — NO section_id FK on painting_sessions
- Denormalized section_name TEXT on painting_sessions (matches detachment_name, weapon_name pattern)
- computeWorkflowPosition is pure (no React/DB deps) — handles orphaned step IDs gracefully
- useWorkflowPositions follows batch enrichment pattern (sorted IDs, Map result, 5min staleTime)
- Design deviation D-08: execution_mode as dot-separated text instead of Badge (RUI-03 partial)

**Issues Resolved:**
- Stale workflow-positions cache after session creation (missing invalidation in useCreatePaintingSession)
- Missing VERIFICATION.md for phases 59 and 60
- 5 unchecked REQUIREMENTS.md boxes despite implementation being complete

**Issues Deferred:**
- RUI-03 partial: execution_mode displayed as text not Badge — intentional design choice (D-08)

**Technical Debt Incurred:**
- 3 orphaned type aliases (SectionType, Technique, ExecutionMode) exported but unused — forward-compatible
