# HobbyForge v3.1 — Recipes 3.0: Hierarchical Painting Workflows

## Purpose

v2.5 upgraded recipes from a flat paint-link list to structured steps with painting phases, tools, techniques, dilution, time estimates, photos, and substitute paints. The current model is:

```text
Recipe (21 columns — name, faction, unit, style, surface, difficulty, etc.)
  Step (15 columns — step_name, paint_id, painting_phase, tool, technique, dilution, time, photo, alt_paint, etc.)
  Step
  Step
```

This still doesn't model real painting workflows. Hobbyists don't always paint every basecoat first, then every shade, then every layer. The correct process is often to complete one surface or technique block before moving to the next.

Example: Ultramarines armor with sponge weathering requires completing the entire blue armor workflow before moving to trim, weapons, lenses, or basing.

The target model is:

```text
Recipe
  Section / Workflow
    Step
    Step
  Section / Workflow
    Step
    Step
```

This allows HobbyForge to represent realistic painting sequences — one surface, technique block, or subassembly at a time.

## Product goal

Turn recipes into a flexible workflow system that supports both simple and advanced painting methods:

- Completing one surface before another
- Completing one technique block before another
- Painting subassemblies separately
- Adding optional display-level upgrade sections
- Tracking recipe progress through Log Session later

---

## Non-negotiable constraints

1. Do not destroy existing recipe data — migration must be additive
2. Existing `recipe_steps` data must migrate safely into default sections
3. Keep existing recipe flows working during transition
4. Preserve TypeScript strict mode
5. Preserve sibling portal architecture
6. Do not duplicate metadata already on recipe or step level
7. Keep the UI compact — sections should not make the form overwhelming
8. Prefer ALTER TABLE over new tables where possible

## Target concept

### Level 1 — Recipe (exists)

Already has 21 columns with full metadata (style, surface, effect, difficulty, estimated_minutes, result_photo_path).

### Level 2 — Section (new)

A coherent surface, technique block, subassembly, or effect grouping.

Examples:
- Armor Blue
- Gold Trim
- Weapon Casing
- Lenses
- Base
- Sponge Weathering
- Airbrush Stage

### Level 3 — Step (exists, gains section_id)

Already has 15 columns with full metadata (painting_phase, tool, technique, dilution, time_estimate_minutes, step_photo_path, alt_paint_id). Gains a nullable `section_id` FK.

## Proposed database changes

### Migration 016 — recipe_sections + recipe_steps.section_id

```sql
-- New table: recipe sections (workflow groupings)
CREATE TABLE IF NOT EXISTS recipe_sections (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    recipe_id     INTEGER NOT NULL REFERENCES painting_recipes(id) ON DELETE CASCADE,
    name          TEXT NOT NULL,
    surface       TEXT,
    optional      INTEGER NOT NULL DEFAULT 0,
    order_index   INTEGER NOT NULL DEFAULT 0,
    notes         TEXT,
    created_at    TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at    TEXT NOT NULL DEFAULT (datetime('now'))
);

-- Add section FK to existing recipe_steps
ALTER TABLE recipe_steps ADD COLUMN section_id INTEGER REFERENCES recipe_sections(id) ON DELETE CASCADE;
```

**Design rationale — slim sections table:**
- `surface` is the one metadata field that adds genuine value at the section level (groups steps by area of the miniature)
- `optional` marks display-level upgrade sections the user can skip for tabletop quality
- No `technique`, `tool`, or `execution_mode` at section level — steps already carry this per-step, and duplicating it creates confusion about which value is authoritative
- No `depends_on_step_id` — dependency graphs are out of scope

### Data migration (within same 016 migration)

```sql
-- Create one default section per existing recipe
INSERT INTO recipe_sections (recipe_id, name, order_index)
SELECT id, COALESCE(NULLIF(area, ''), 'General'), 0
FROM painting_recipes;

-- Point existing steps at their recipe's default section
UPDATE recipe_steps
SET section_id = (
    SELECT rs.id FROM recipe_sections rs
    WHERE rs.recipe_id = recipe_steps.recipe_id
    LIMIT 1
);
```

### Backward compatibility

- `section_id` on recipe_steps is nullable — steps without a section still work
- UI reads sections when they exist; falls back to flat step list if none
- Existing recipe creation creates one default section automatically

## UI changes

### Replace flat step list with sectioned layout

Current:
```text
RecipeStepList
  RecipeStepRow
  RecipeStepRow
```

Target:
```text
RecipeSectionList
  RecipeSectionCard (collapsible)
    RecipeStepList
      RecipeStepRow
    + Add Step button
  RecipeSectionCard
    RecipeStepList
      RecipeStepRow
  + Add Section button
```

### New components

- `src/features/recipes/RecipeSectionList.tsx` — list of section cards with section-level drag-and-drop
- `src/features/recipes/RecipeSectionCard.tsx` — collapsible card showing section name, surface, step count, missing paint count
- `src/features/recipes/RecipeSectionForm.tsx` — inline section name/surface editor

### Updated components

- `RecipeFormSheet.tsx` — draft sections containing draft steps (instead of flat draft steps)
- `RecipeDetailSheet.tsx` — workflow timeline grouped by section
- `RecipeStepTimeline.tsx` — section headers between step groups
- `recipeSteps.ts` — DraftSection type alongside existing DraftStep

### Recipe form UX principles

- Recipe-level fields remain at the top (unchanged)
- Sections are collapsible cards within the form
- Each section shows: name, surface badge, step count, optional badge
- Steps inside a section use existing RecipeStepRow (unchanged)
- Creating a simple one-section recipe must be just as easy as today
- Default section auto-created so the user doesn't need to think about sections for simple recipes

### Recipe detail UX

RecipeDetailSheet becomes a workflow timeline:

```text
Armor Blue (Armor) — 7 steps, ~45 min
  1. Prime black          ■ Chaos Black (owned)
  2. Sponge dark blue     ■ Kantor Blue (owned)
  3. Sponge mid blue      ■ Caledor Sky (missing!)
  ...

Gold Trim (Metal) — 4 steps, ~20 min
  1. Basecoat dark bronze  ■ Balthasar Gold (owned)
  ...
```

## Acceptance criteria

1. Recipes support multiple workflow sections
2. Each section contains ordered steps
3. Existing recipe data is preserved and migrated (zero data loss)
4. User can model "complete all armor steps before trim" workflows
5. Simple one-section recipes are just as easy to create as before
6. Missing/owned paint indicators still work
7. Recipe detail displays a clear workflow timeline grouped by section
8. Drag-and-drop ordering works within sections and between sections
9. TypeScript build passes with strict mode
10. Tests cover migration, queries, and core UI behavior
11. Recipe duplication copies sections

## Out of scope

- Per-step completion history / progress tracking
- Moving steps between sections via drag-and-drop (reorder within section only for v1)
- Section-level execution mode (batch_step_across_models, model_by_model)
- Step dependency graph (depends_on_step_id)
- AI-assisted recipe generation
- Per-section photos
- LogSessionSheet section selector (future integration)

---

## Product principle

The system should not force hobbyists into a global basecoat/layer/highlight sequence. It should support real painting behavior:

> Sometimes the right workflow is to finish one surface, one technique block, or one subassembly completely before moving to the next part of the miniature.

HobbyForge recipes should become flexible painting workflows, not just paint checklists.
