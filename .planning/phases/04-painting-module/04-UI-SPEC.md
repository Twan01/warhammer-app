## Spacing Scale

Declared values (multiples of 4 — inherited from Phase 3):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge padding, grip handle padding |
| sm | 8px | Card internal padding (vertical), button group gaps, step row gaps |
| md | 16px | Card internal padding (horizontal), form field gaps, sheet padding |
| lg | 24px | Column header to card list gap, section spacing within recipe sheet |
| xl | 32px | Board horizontal gap between columns |
| 2xl | 48px | Empty state vertical padding (py-12) |
| 3xl | 64px | Not used in this phase |

Exceptions (Phase 4 specific):
- Kanban card minimum width: 280px — prevents card text from wrapping destructively at default column width
- Kanban column width: 280px fixed (horizontal scroll on the board if columns exceed viewport)
- Kanban card min height: 44px touch/click target on all interactive card elements
- Drag handle touch target: 24px × 24px (h-6 w-6) to give adequate pointer area on grip icons

Source: CONTEXT.md §Kanban card density; Phase 3 UI-SPEC spacing scale.

## Typography

Inherited from Phase 3 (unchanged):

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px (text-sm) | 400 (normal) | 1.5 |
| Body variant / secondary text | 12px (text-xs) | 400 (normal) | 1.4 |
| Page heading | 20px (text-xl) | 600 (semibold) | 1.2 |
| Detail sheet field heading / column header label | 12px (text-xs) | 600 (semibold) | 1.2 |

Rules (inherited, apply to all Phase 4 components):
- Exactly 2 weights: 400 (normal) and 600 (semibold). Weight 500 / `font-medium` is NOT used.
- Kanban column heading: 12px / weight 600 / uppercase / tracking-wide — `text-xs font-semibold text-muted-foreground uppercase tracking-wide`
- Kanban card unit name: 14px / weight 600 (semibold) — promoted from body because the card name is the primary identifier within a compact card
- Recipe table columns: same rules as UnitTable column headers (text-xs font-semibold uppercase tracking-wide)
- SheetTitle for recipe detail/edit: 16px / semibold — inherits shadcn Sheet default

Source: Phase 3 UI-SPEC §Typography (inherited); CONTEXT.md §Kanban card density.

## Color

All values reference the dark-mode CSS variables from `src/styles/globals.css` (`.dark` block). App is dark-mode-only at runtime.

Inherited from Phase 3 (unchanged):

| Role | CSS Variable | Dark Value | Usage |
|------|-------------|------------|-------|
| Dominant (60%) | `--background` | hsl(240 10% 3.9%) — near-black slate | Page background, Kanban board background, recipe table rows |
| Secondary (30%) | `--card` / `--sidebar` | hsl(240 10% 5.9%) — dark zinc | Kanban cards, recipe detail Sheet, filter bar, popover background |
| Accent (10%) | `faction.color_theme` hex | User-defined per faction | Faction badge on Kanban card and recipe table row ONLY |
| Destructive | `--destructive` | hsl(0 72.2% 50.6%) — red | Delete button, delete dialog confirm button, overdue target date text |
| Muted | `--muted-foreground` | hsl(240 5% 64.9%) | Secondary card text, column labels, empty state copy, drag handle icon |

Phase 4 additions:
| Role | Value | Usage |
|------|-------|-------|
| Paint owned indicator | `text-green-500` (Tailwind green-500, ~hsl(142 71% 45%)) | Owned paint badge / indicator dot on recipe step |
| Paint missing indicator | `text-red-500` (Tailwind red-500, ~hsl(0 72% 51%)) | Unowned/missing paint badge / indicator dot on recipe step |
| Overdue target date | `text-destructive` (var(--destructive)) | Target completion date text on Kanban card when date < today |

Accent reserved for:
1. Faction badge background on Kanban cards (`<Badge style={{ backgroundColor: faction.color_theme }}>`) — PROJ-03
2. Faction badge background in recipe table rows and recipe detail Sheet — RECIPE-07 / POLISH-05

The accent is NOT used for: drag-and-drop active state (use `bg-accent` CSS variable instead), column hover, recipe step rows, progress bar fill, or any generic interactive element.

Source: Phase 3 UI-SPEC §Color (inherited); CONTEXT.md §Kanban card density; REQUIREMENTS.md RECIPE-06, PROJ-03.

## Visual Hierarchy

**Painting Projects page:** KanbanBoard is the primary focal point. Page heading "Painting Projects" anchors top-level hierarchy. Column headings are secondary (status names). Cards are tertiary. Card action menus are quaternary.

**Recipes page:** RecipeTable is the primary focal point, matching the Collection page pattern. Page heading "Recipes" anchors top-level hierarchy. Filter bar is secondary. Recipe detail Sheet is a tertiary overlay layer.

Both pages follow the same two-tier hierarchy already established: page heading → primary content surface → right-side detail Sheet.

## Interaction Contracts

### KanbanBoard (PROJ-01, PROJ-02, PROJ-07)

- Layout: horizontal flex row of columns, `gap-8 (32px)` between columns, full-height scroll container
- Each column: 280px fixed width, `flex flex-col gap-2`
- Column header: status label (from `PAINTING_STATUS_ORDER`) at `text-xs font-semibold uppercase tracking-wide text-muted-foreground`, plus a unit count badge (`<Badge variant="secondary">N</Badge>`) inline right
- Column body: vertically scrollable (`overflow-y-auto`), with cards stacked at `gap-2 (8px)`
- Empty column behavior: column collapses (not rendered) — only columns with ≥1 active unit shown (CONTEXT.md decision)
- Cards within a column sorted by: priority ASC first, then target_completion_date ASC (nulls last) — PROJ-07
- "Not Started" column: included in the board as the entry column for newly-active units

### KanbanCard (PROJ-03)

Card layout (top to bottom, 16px internal horizontal padding, 12px vertical padding):

1. **Row 1 — Header:** `[GripVertical icon] [Unit name] [· · · menu trigger]`
   - GripVertical: `h-4 w-4 text-muted-foreground cursor-grab`, left-aligned, drag handle for @dnd-kit
   - Unit name: `text-sm font-semibold flex-1`, truncated at 1 line (`truncate`)
   - Menu trigger: `MoreHorizontal` icon button (h-4 w-4), right-aligned, opens card action DropdownMenu

2. **Row 2 — Faction badge:** `<Badge style={{ backgroundColor: faction.color_theme }} className="text-xs">` faction name — spans full width if needed

3. **Row 3 — Progress bar + percentage:** `<Progress value={painting_percentage} className="h-1.5" />` with `<span className="text-xs text-muted-foreground">{painting_percentage}%</span>` right-aligned inline

4. **Row 4 — Meta row (conditional):** `flex items-center gap-2 text-xs text-muted-foreground`
   - Priority indicator: shown only if priority is set — `Flag` icon (h-3 w-3) + priority text
   - Target date: shown only if target_completion_date is set — `Calendar` icon (h-3 w-3) + date formatted as "MMM D, YYYY"; date text uses `text-destructive` class if date is before today

Card container: `<Card className="cursor-grab active:cursor-grabbing select-none">`, `rounded-lg`, background `bg-card`
Card must carry `aria-label={unit.name}` on the draggable wrapper for screen reader context.

### Card Action Menu (PROJ-06)

DropdownMenu anchored to the MoreHorizontal trigger:

| Item | Icon | Action |
|------|------|--------|
| "Remove from board" | `X` (h-4 w-4) | Sets `is_active_project = false` optimistically; card disappears |
| "Edit unit" | `Pencil` (h-4 w-4) | Opens UnitSheet in edit mode for the unit |

No delete action on Kanban card — deletion lives in Collection page and unit detail Sheet only.

### Drag-and-Drop (PROJ-04, PROJ-05)

- @dnd-kit/core `DndContext` wraps the full board
- @dnd-kit/sortable `SortableContext` per column with `verticalListSortingStrategy`
- Drag overlay: a translucent clone of the dragged card at 90% opacity (`opacity-90`) rendered in the `DragOverlay` portal
- Drop target column highlights: `bg-accent/30` on the column body when a card is dragged over it
- On drop: call `useUpdateUnit` with `{ status_painting: targetColumnStatus }` — optimistic `setQueryData` before mutation, rollback in `onError` with Sonner toast "Status update failed. The card has been moved back."
- Drag cancellation (Escape key): card snaps back to origin column — @dnd-kit handles this natively

### Add Project Picker (PROJ-06)

- "Add project" button on the Kanban page header: `<Button variant="default" size="sm">Add project</Button>`
- Opens a Popover + Command picker (same pattern as CategoryCombobox) listing units where `is_active_project = 0`, searchable by name
- CommandEmpty: "No inactive units found."
- Selecting a unit: sets `is_active_project = true` optimistically, closes popover, card appears in the unit's current status column
- The picker must re-fetch or filter from the cached units list — no separate query needed

### Kanban Loading State (POLISH-02)

- While `useUnits()` `isLoading`: render 3 skeleton columns — each column shows 3 `<Skeleton className="h-24 w-[280px] rounded-lg" />` cards
- Page heading remains visible during loading

### Kanban Empty State (PROJ-08)

- Shown when no units have `is_active_project = true`
- Layout: `flex flex-col items-center justify-center gap-4 py-16 text-center` — matches FactionsEmptyState.tsx pattern
- Icon: `Kanban` (lucide-react, h-12 w-12, `text-muted-foreground`)
- Heading: "No active projects" — 14px / semibold
- Body: "Mark a unit as active project to see it here."
- CTA button: `variant="default"` — "Add project" (opens the same unit picker popover)

### RecipeTable (RECIPE-07)

Table layout mirrors UnitTable pattern (TanStack Table into shadcn Table):

Columns in order: Recipe name, Faction (badge), Linked Unit, Area, Step count, Actions

| Column | Width | Notes |
|--------|-------|-------|
| Recipe name | flex-grow | Truncated at 1 line |
| Faction | 120px | Colored Badge, or em-dash if unlinked |
| Linked Unit | 140px | Unit name, or em-dash if unlinked |
| Area | 120px | Free text (e.g. "Armor", "Skin") |
| Step count | 80px | `N steps` text-xs text-muted-foreground |
| Actions | 80px | Edit (Pencil icon) + Delete (Trash2 icon) icon buttons |

- Row click opens the recipe detail Sheet
- Sortable: Recipe name ASC default; faction and area columns also sortable
- No pagination in Phase 4 (recipe count expected to be small — add pagination in a future phase if needed)
- Filters above table: Faction (Popover multi-select, same as Collection), Linked Unit (Popover single-select), Area (Input text filter)

### Recipe Detail Sheet (RECIPE-01 through RECIPE-06)

- `side="right"`, `key={selectedRecipe.id}` — prevents stale data (POLISH-04)
- SheetHeader: Recipe name (SheetTitle) + Faction badge inline
- Content sections (Separator between each):
  1. **Meta:** Linked unit name (or "Not linked"), Area, tutorial_link as `<a href={...} target="_blank" rel="noopener noreferrer" className="text-sm text-primary underline">` clickable link, Notes
  2. **Recipe Steps:** ordered list; each step row: `[N. step_name] [paint indicator dot] [paint name]`
     - Paint owned: `●` circle dot in `text-green-500` before paint name
     - Paint missing / unowned: `●` circle dot in `text-red-500` before paint name
     - Step with no linked paint: step name only, no dot
- Footer: "Edit Recipe" button (`variant="default"`) + "Delete Recipe" button (`variant="ghost" text-destructive`)

### Recipe Edit/Create Sheet (RECIPE-01 through RECIPE-05)

- Opens as a stacked Sheet (same pattern as UnitSheet from Phase 2) — `side="right"`, opens on top of detail Sheet when editing from detail
- `key={recipe?.id ?? "new"}` on SheetContent — POLISH-04
- Form fields (top to bottom):
  1. Recipe name — `<Input>`, required
  2. Faction — `<Select>` from factions list, optional
  3. Linked Unit — `<Select>` filtered by selected faction (shows all units if no faction selected), optional
  4. Area — `<Input placeholder="e.g. Armor, Skin, Base...">`, free-text, optional
  5. Tutorial link — `<Input type="url" placeholder="https://...">`, optional
  6. Notes — `<Textarea>` (not a shadcn component — use `<textarea className="...">` or add Textarea via shadcn if installed), optional
  7. **Steps section** — labeled "Recipe Steps" with a `+` icon button to add a step:
     - Each step row: `[GripVertical drag handle] [step name Input] [paint Combobox] [Trash2 remove button]`
     - Step name Input: placeholder suggestions "Primer", "Basecoat", "Shade", "Layer", "Highlight", "Glaze", "Weathering", "Technical", "Basing" — shown as CommandSuggestion items in a Command dropdown or as `datalist` (use `datalist` for simplicity; no extra Popover overhead)
     - Paint Combobox: Popover + Command, same pattern as CategoryCombobox — searches paint inventory by `brand + name`, shows `[brand] [name]` in results, optional per step
     - Steps are reorderable via @dnd-kit drag handles (same @dnd-kit already used for Kanban)
     - Removing a step: immediate (no confirm dialog); step removal is non-destructive until form submit
- Footer: Submit button (`"Save Recipe"` edit mode, `"Add Recipe"` create mode) + Cancel button `variant="outline"`
- Submit button `disabled={isSubmitting}`

### RecipePaintLinker — Owned/Missing Indicator (RECIPE-06, PAINT-03, PAINT-04)

- In the recipe detail Sheet: each step's linked paint shows the colored dot (green owned / red missing) — no interactive element, read-only
- In the recipe edit form: the paint Combobox search results show the same colored dot inline with each paint name in the dropdown list — `●` in `text-green-500` or `text-red-500` based on `paint.owned` field
- Inline paint create inside recipe builder (PAINT-03): "Add new paint" option appears as a CommandItem at the bottom of the paint Combobox results — clicking opens a minimal PaintSheet (same as PaintsPage) in a stacked layer; on save, the new paint auto-selects in the step's combobox

### Recipe Loading State (POLISH-02)

- While `useRecipes()` `isLoading`: render 5 skeleton rows — `<Skeleton className="h-10 w-full" />` — in place of table body

### Recipe Empty State (RECIPE-08)

- Shown when no recipes exist
- Layout: same pattern as FactionsEmptyState.tsx (`flex flex-col items-center justify-center gap-4 py-16 text-center`)
- Icon: `BookOpen` (lucide-react, h-12 w-12, `text-muted-foreground`)
- Heading: "No recipes yet"
- Body: "Document your paint schemes. Add a recipe to get started."
- CTA: `<Button variant="default">Add Recipe</Button>`

### Recipe Delete Confirm (RECIPE-04, POLISH-01)

- AlertDialog pattern (same as FactionDeleteDialog.tsx / UnitDeleteDialog.tsx)
- Dialog title: "Delete recipe?"
- Body: "This will permanently remove [recipe name] and all its steps. This cannot be undone."
- Confirm button: "Delete recipe" (`variant="destructive"`)
- Cancel: "Cancel"

### Unit Detail Sheet — Recipes Section (CONTEXT.md §Recipe ↔ Unit linkage)

- Added as a new read-only section at the bottom of the Phase 3 unit detail Sheet
- Section heading: "Linked Recipes" (text-xs font-semibold uppercase tracking-wide text-muted-foreground)
- Content: list of recipe names as `<Button variant="link" size="sm">` — clicking navigates to `/recipes` with that recipe's detail Sheet open
- Empty: "No recipes linked to this unit." in text-xs text-muted-foreground
