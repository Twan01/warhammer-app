## Typography

Inherit from project-wide type system (globals.css, shadcn new-york). Consistent with Phase 9 PlaybookTab contract.

| Role | Size | Weight | Line Height | Usage in Phase 15 |
|------|------|--------|-------------|-------------------|
| Body | 14px (`text-sm`) | 400 (`font-normal`) | 1.5 | Ability description text, Sources list, picker search results, sync date label |
| Label | 12px (`text-xs`) | 600 (`font-semibold`) | 1 | Section headings ("Datasheet Abilities", "Sources", "Personal Ability Notes"), sub-group labels ("Core Abilities", "Faction Abilities", "Unit Abilities"), stat abbreviations, form field labels |
| Heading | 16px (`text-base`) | 600 (`font-semibold`) | 1.2 | Collapsible trigger label ("Datasheet Abilities"), ability name within each ability entry |
| Display | Not used in this phase | — | — | — |

Section label pattern (inherited from Phase 9): `text-xs font-semibold text-muted-foreground uppercase tracking-wide`

Sub-group label pattern (new in Phase 15): `text-xs font-semibold text-muted-foreground uppercase tracking-wide` — same as section label; differentiated by indentation (`pl-0`) vs ability rows (`pl-2`)

Ability name pattern: `text-sm font-semibold text-foreground` — distinct from description body text

Ability description pattern: `text-sm font-normal text-muted-foreground leading-relaxed` — readable multi-line text

### New: Datasheet Abilities Collapsible

Component: inline in `PlaybookTab.tsx` using shadcn `Collapsible`.

```
<Collapsible defaultOpen={true}>
  <CollapsibleTrigger asChild>
    <button
      className="flex items-center justify-between w-full py-2 text-left"
    >
      <span className="text-base font-semibold">Datasheet Abilities</span>
      <ChevronDown
        className="h-4 w-4 text-muted-foreground transition-transform data-[state=open]:rotate-180"
      />
    </button>
  </CollapsibleTrigger>
  <CollapsibleContent>
    <div className="flex flex-col gap-4">
      {/* Sub-group: Core Abilities */}
      {coreAbilities.length > 0 && (
        <div className="flex flex-col gap-2">
          <span className={SECTION_LABEL_CLASS}>Core Abilities</span>
          {coreAbilities.map(a => <AbilityEntry key={a.id} ability={a} />)}
        </div>
      )}
      {/* Sub-group: Faction Abilities */}
      {factionAbilities.length > 0 && (
        <div className="flex flex-col gap-2">
          <span className={SECTION_LABEL_CLASS}>Faction Abilities</span>
          {factionAbilities.map(a => <AbilityEntry key={a.id} ability={a} />)}
        </div>
      )}
      {/* Sub-group: Unit Abilities */}
      {unitAbilities.length > 0 && (
        <div className="flex flex-col gap-2">
          <span className={SECTION_LABEL_CLASS}>Unit Abilities</span>
          {unitAbilities.map(a => <AbilityEntry key={a.id} ability={a} />)}
        </div>
      )}
    </div>
  </CollapsibleContent>
</Collapsible>
```

`<AbilityEntry>` is a module-local sub-component:
```tsx
function AbilityEntry({ ability }: { ability: RwDatasheetAbility }) {
  return (
    <div className="flex flex-col gap-1 pl-2 border-l border-border">
      <span className="text-sm font-semibold text-foreground">{ability.name}</span>
      {ability.description && (
        <p className="text-sm text-muted-foreground leading-relaxed">
          {ability.description}
        </p>
      )}
    </div>
  );
}
```

`defaultOpen={true}` — collapsible starts expanded so the imported data is immediately visible.

ChevronDown rotation on open/close: `data-[state=open]:rotate-180` (Collapsible sets `data-state` on trigger automatically via Radix).

### New: DatasheetImportDialog (conflict resolution)

Component: `src/features/units/DatasheetImportDialog.tsx`

Mounted as a sibling to UnitDetailSheet in CollectionPage.tsx (same portal pattern as lightbox in Phase 13, army list dialog in Phase 8).

```
<Dialog open={open} onOpenChange={...}>
  <DialogContent className="sm:max-w-lg max-h-[80vh] overflow-y-auto">
    <DialogHeader>
      <DialogTitle>Review Import</DialogTitle>
      <DialogDescription>
        Some fields already have values. Choose which to keep for each field.
      </DialogDescription>
    </DialogHeader>
    <div className="flex flex-col divide-y divide-border">
      {conflicts.map(field => (
        <div key={field.key} className="flex items-center justify-between py-3 gap-4">
          <div className="flex flex-col gap-0.5 flex-1 min-w-0">
            <span className="text-xs font-semibold text-muted-foreground uppercase">
              {field.label}
            </span>
            <div className="flex gap-2 text-sm">
              <span className="text-muted-foreground truncate">
                Yours: {field.currentValue || "—"}
              </span>
              <span className="text-foreground truncate">
                Datasheet: {field.incomingValue}
              </span>
            </div>
          </div>
          <div className="flex gap-1 shrink-0">
            <Button
              type="button"
              size="sm"
              variant={field.choice === "keep" ? "default" : "outline"}
              onClick={() => setChoice(field.key, "keep")}
            >
              Keep
            </Button>
            <Button
              type="button"
              size="sm"
              variant={field.choice === "use" ? "default" : "outline"}
              onClick={() => setChoice(field.key, "use")}
            >
              Use datasheet
            </Button>
          </div>
        </div>
      ))}
    </div>
    <DialogFooter>
      <Button variant="ghost" onClick={onClose}>Discard changes</Button>
      <Button variant="default" onClick={handleConfirm}>Apply import</Button>
    </DialogFooter>
  </DialogContent>
</Dialog>
```

Default choice for each field: "use" (datasheet value wins unless user actively selects "Keep"). This is the expected happy-path — the point of importing is to use the datasheet data.

"Apply import" loads selected values into the PlaybookTab form. Existing Save Playbook button commits to DB. This is consistent with Phase 9 manual edit behavior.

### Sync Trigger Location

The sync action is accessible via two paths (Claude's Discretion resolved):
1. The "Sync now" inline link in the empty-banner inside PlaybookTab (primary path when DB is empty)
2. A "Re-sync" icon button (`RefreshCw` from lucide-react, `variant="ghost" size="icon"`) placed in the PlaybookTab header area next to "Stats", visible only when `rules.db` is populated and a datasheet is linked

Sync does NOT have a separate settings page entry in Phase 15. Sync is contextual to the Playbook tab.

During sync: show a loading state on the Re-sync/Sync button (`disabled + Loader2 spin icon`). No full-page overlay. On success: `toast.success("Datasheets synced")`. On failure: inline error below the banner or `toast.error("Sync failed — check your connection and try again")`.

## Interaction Contract

### Sync Flow

1. User opens PlaybookTab on any unit. `useRulesSyncMeta()` resolves.
2. If `rules.db` is empty (sync meta returns null): banner renders inline; no import controls shown.
3. User clicks "Sync now" in banner OR re-sync button: `useRulesSync().mutate()` fires.
4. During sync: button shows `Loader2` spinning icon, `disabled`. No other UI blocked.
5. On success: `toast.success("Datasheets synced")`. Banner disappears. Import controls appear.
6. On error: `toast.error("Sync failed — check your connection and try again")`. Banner remains.

### Picker Auto-Trigger

On PlaybookTab mount, check:
- Does `unit_strategy_notes.datasheet_id` have a value? → No auto-picker if yes.
- Is `rules.db` populated (sync meta exists)? → No auto-picker if no.
- Do all 6 stats (M, T, Sv, W, Ld, OC) have null values? → Auto-open picker if yes.

If all three conditions are met: `DatasheetPicker` opens automatically without button click.

If any condition fails: picker only opens on "Import stats" button click.

### Picker Interaction

1. Picker opens with focus on search input.
2. List pre-filtered to unit's faction (all datasheets for that faction from `rules.db`).
3. User types: client-side filter by `name` (case-insensitive substring match).
4. User clicks a datasheet row: picker closes, import begins immediately.
5. If unit already has data in any field: conflict dialog opens (sibling portal).
6. If unit has no data in any field: values load directly into PlaybookTab form, no dialog.
7. User clicks "Skip": picker closes, nothing happens. Import controls remain available.
8. Escape key or clicking outside: same as Skip.

### Conflict Resolution Flow

1. `DatasheetImportDialog` receives the list of conflicting fields (only fields where both current and incoming values are non-empty).
2. All fields default to `choice: "use"` (datasheet value).
3. User reviews, optionally toggles individual fields to "Keep".
4. "Apply import": selected values load into PlaybookTab local state (same `setMove`, `setToughness`, etc. calls as manual edit). Dialog closes.
5. "Discard changes": dialog closes, nothing changes.
6. After Apply import: `isDirty` becomes true. User must click "Save Playbook" to persist.

Non-conflicting fields (where current value is null/empty): always loaded from datasheet without appearing in dialog.

### Datasheet Abilities Collapsible

- Starts expanded (`defaultOpen={true}`) — imported data should be visible immediately.
- Toggle: click the trigger to collapse/expand. `ChevronDown` rotates 180deg when open.
- Sub-groups render only if they have at least one ability (empty sub-groups are hidden).
- If all sub-groups are empty (datasheet has no abilities): hide the entire Datasheet Abilities section (no empty collapsible shown). This is rare but possible.

### Multi-Profile Handling

- Line 1 of `rw_datasheet_models` populates the stats block (M/T/Sv/W/Ld/OC).
- If `MAX(line) > 1`: display the multi-profile note below the stats grid: "Additional model profiles available — see Datasheet Abilities for details."
- Additional model profiles are NOT displayed as a second stats grid.
