Create `src-tauri/migrations/018_recipe_sections.sql` with exactly four SQL statements:

**Step 1 — CREATE TABLE recipe_sections:**
```sql
CREATE TABLE IF NOT EXISTS recipe_sections (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    recipe_id   INTEGER NOT NULL REFERENCES painting_recipes(id) ON DELETE CASCADE,
    name        TEXT NOT NULL DEFAULT 'Steps',
    surface     TEXT,
    optional    INTEGER NOT NULL DEFAULT 0,
    order_index INTEGER NOT NULL DEFAULT 0,
    notes       TEXT,
    created_at  TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
```

Column details:
- `id`: AUTOINCREMENT primary key
- `recipe_id`: NOT NULL FK to painting_recipes(id), ON DELETE CASCADE (section dies with recipe)
- `name`: NOT NULL, DEFAULT 'Steps' (neutral default for auto-created sections)
- `surface`: nullable TEXT (same domain as painting_recipes.surface: Armour, Cloth, Skin, Weapons, Base etc.)
- `optional`: NOT NULL INTEGER DEFAULT 0 (SQLite 0|1 boolean; 0 = required, 1 = skippable)
- `order_index`: NOT NULL INTEGER DEFAULT 0 (display order; no UNIQUE constraint — reorder loop would conflict)
- `notes`: nullable TEXT
- `created_at`/`updated_at`: TEXT with datetime('now') defaults

**Step 2 — ALTER TABLE recipe_steps:**
```sql
ALTER TABLE recipe_steps ADD COLUMN section_id INTEGER REFERENCES recipe_sections(id) ON DELETE CASCADE;
```
Nullable FK — ON DELETE CASCADE means steps die when their section is deleted. The cascade chain continues: step deletion triggers ON DELETE SET NULL on painting_sessions.recipe_step_id (from migration 014), clearing session links without deleting sessions.

**Step 3 — Data migration INSERT:**
```sql
INSERT INTO recipe_sections (recipe_id, name, order_index, created_at, updated_at)
SELECT id, 'Steps', 0, datetime('now'), datetime('now')
FROM painting_recipes;
```
Creates exactly one default section named "Steps" for every existing recipe. No duplicates possible because this runs once at migration time.

**Step 4 — Data migration UPDATE:**
```sql
UPDATE recipe_steps
SET section_id = (
    SELECT id FROM recipe_sections
    WHERE recipe_sections.recipe_id = recipe_steps.recipe_id
    LIMIT 1
)
WHERE section_id IS NULL;
```
Points every existing step at its recipe's newly-created default section. The correlated subquery is guaranteed to find a match because Step 3 just inserted one section per recipe. After this statement, zero steps should have section_id IS NULL.

```typescript
/**
 * RecipeSection entity (v0.2.7 SECT-01).
 * Mirrors the recipe_sections table created in migration 018.
 * A section groups recipe steps into logical workflow blocks
 * (e.g., Armour, Cloth, Weapons, Basing).
 */
export interface RecipeSection {
  id: number;
  recipe_id: number;
  name: string;
  surface: string | null;
  optional: number;          // 0 | 1 SQLite boolean — 0 = required, 1 = skippable
  order_index: number;
  notes: string | null;
  created_at: string;
  updated_at: string;
}

export type CreateRecipeSectionInput = Omit<RecipeSection, "id" | "created_at" | "updated_at">;
export type UpdateRecipeSectionInput = Partial<Omit<CreateRecipeSectionInput, "recipe_id">> & { id: number };
```

Fields:
- `id`: number (AUTOINCREMENT PK)
- `recipe_id`: number (FK to painting_recipes)
- `name`: string (NOT NULL, default "Steps")
- `surface`: string | null (same domain vocabulary as painting_recipes.surface)
- `optional`: number (0|1 SQLite boolean discipline)
- `order_index`: number (display position)
- `notes`: string | null
- `created_at`/`updated_at`: string (ISO datetime from SQLite)

CreateRecipeSectionInput omits `id`, `created_at`, `updated_at` (DB-generated).
UpdateRecipeSectionInput omits `recipe_id` (immutable after creation), makes all remaining fields partial, requires `id`.
