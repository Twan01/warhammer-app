## Component Inventory

### 1. PointsSourceChip (new component)

**Location:** `src/features/army-lists/PointsSourceChip.tsx`
**Purpose:** Displays resolved points value with source provenance label (PV-02).

**Visual spec:**
- Layout: inline-flex, items-center, gap-1.5
- Colored dot: 6x6 rounded-full (`h-1.5 w-1.5`), color varies by source
- Text: 12px medium weight, `text-muted-foreground`
- Format examples:
  - `[green dot] 95 pts . synced`
  - `[violet dot] 100 pts . override`
  - `[amber dot] 80 pts . user-override`
  - `[blue dot] 75 pts . base`
  - `[muted dot] -- . unknown`

**Source-to-color mapping:**

| Source | Dot class | Label |
|--------|-----------|-------|
| `"synced"` | `bg-emerald-500` | `synced` |
| `"override"` | `bg-violet-500` | `override` |
| `"user-override"` | `bg-amber-500` | `user-override` |
| `"base"` | `bg-blue-500` | `base` |
| `"unknown"` | `bg-muted-foreground/50` | `unknown` |

**Props:** `{ points: number | null; source: "synced" | "override" | "user-override" | "base" | "unknown" }`

Auto-decision: Reuses the same dot+text pattern as PointsFreshnessBadge and StatusBadge for consistency. The `middle dot` separator matches the CONTEXT.md format "95 pts . synced".

### 2. MatchStatusIndicator (new component)

**Location:** `src/features/army-lists/MatchStatusIndicator.tsx`
**Purpose:** Shows unit-to-rules mapping status inline on unit rows (PV-03, PV-05).

**Visual spec:**
- Layout: inline-flex button (ghost variant), h-6 w-6, rounded
- Clickable: opens RulesMappingSheet on click
- Three states:

| match_status | Icon | Color | Tooltip |
|-------------|------|-------|---------|
| `"confirmed"` | `Check` (lucide) | `text-emerald-500` | "Rules mapping confirmed" |
| `"auto"` | `LinkIcon` (lucide) | `text-muted-foreground` | "Auto-matched -- click to confirm" |
| `"manual"` | `Pencil` (lucide) | `text-blue-500` | "Manually mapped" |
| no mapping | `AlertTriangle` (lucide) | `text-amber-500` | "No rules mapping -- click to resolve" |
| ambiguous | `AlertTriangle` (lucide) | `text-destructive` | "Ambiguous match (N candidates) -- click to resolve" |

**Props:** `{ unitId: number; matchStatus: "auto" | "confirmed" | "manual" | null; ambiguousCount?: number; onClick: () => void }`

Auto-decision: Small icon-only button keeps the unit row compact per CONTEXT.md specifics ("subtle to avoid cluttering"). The AlertTriangle icon is already used in ArmyListUnitRow for warnings, maintaining consistency.

### 3. RulesMappingSheet (new component)

**Location:** `src/features/army-lists/RulesMappingSheet.tsx`
**Purpose:** Sheet for confirming or overriding unit-to-rules mapping (PV-04).

**Visual spec:**
- Uses existing shadcn Sheet (side="right", same width pattern as other app sheets)
- Sheet title: unit name, 16px semibold
- Sheet description: "Confirm or change the rules datasheet mapping for this unit"

**Sections (top to bottom):**

1. **Current Match** section
   - Card with `bg-muted/30` background, rounded-lg, p-4
   - Shows: datasheet name, faction, source (`auto` / `manual` / `confirmed`)
   - "Confirm" primary Button (when status is `auto`) -- saves `match_status = "confirmed"`
   - "Remove Mapping" destructive ghost Button (when mapping exists)

2. **Alternative Matches** section (when ambiguous or user wants to change)
   - Heading: "Available Datasheets" at 14px semibold
   - Search Input with placeholder "Search datasheets..."
   - Scrollable list (max-h-64, overflow-y-auto) of datasheet candidates
   - Each candidate: card-like row with datasheet name, faction name, "Select" outline Button
   - Selected/current match highlighted with `ring-2 ring-primary` border

3. **No Match** empty state (when no rules data synced)
   - Centered layout, muted text
   - Message: "No rules data available. Sync rules data from the Workshop to enable auto-matching."

**Interaction flow:**
- Open: click MatchStatusIndicator on unit row
- Confirm auto-match: click "Confirm Match" button -> upsert with `match_status = "confirmed"` -> close Sheet -> toast "Rules mapping confirmed"
- Select alternative: click "Select" on a candidate -> upsert with `match_status = "manual"` + new `rules_datasheet_id` -> close Sheet -> toast "Rules mapping updated"
- Remove: click "Remove Mapping" -> delete mapping -> close Sheet -> toast "Rules mapping removed"

### 4. ArmyListSummaryBar (modified)

**Modifications for warning split (PV-06, PV-07):**

**Current behavior:** Shows "Warnings: N" with tooltip "N critical, M informational"

**New behavior:**
- List-level warnings rendered as inline alert badges directly in the summary bar
- Format: each list warning gets its own Badge (variant="destructive" for hard, variant="outline" for soft)
- Examples:
  - `[destructive badge] Points exceeded` (when total > limit)
  - `[outline badge] Stale points data` (when freshness is stale/never)
- Warning count tooltip updated: "N list warnings, M unit warnings" (replacing "N critical, M informational")

**Layout change:**
- New row below the freshness + warnings row
- Only renders when `computeListWarnings()` returns non-empty array
- Horizontal flex-wrap with gap-2

Auto-decision: Inline badges in the summary bar are more scannable than a single count tooltip. Uses existing Badge component variants for consistency.

### 5. ArmyListUnitRow (modified)

**Modifications:**

- **Source chip:** Replace the current bare points Input placeholder display with PointsSourceChip component, positioned below the points override Input
- **Match indicator:** Add MatchStatusIndicator to the left of the unit name (between the existing warning icon and the name text)
- **Warning removal:** Remove list-level "Points exceeded" hard warning from per-unit computeUnitWarnings output (moved to summary bar via computeListWarnings)

**Column layout (updated):**
1. Warning icon (existing) + Match status icon (new) + Unit name + loadout + role
2. Painting status badge (existing)
3. Points override Input (existing) + PointsSourceChip (new, below Input)
4. Notes toggle (existing)
5. Remove button (existing)

## Interaction Patterns

### Source Chip Display
- Read-only display, no interaction on the chip itself
- Source is computed by `resolveUnitPoints()` pure function from row column values
- No extra queries needed (D-02: intermediate columns exposed in existing SQL)

### Match Status Flow
1. Unit row shows MatchStatusIndicator icon
2. Click icon -> RulesMappingSheet opens
3. User either confirms auto-match OR selects alternative OR removes mapping
4. Sheet closes -> toast notification -> unit row indicator updates via React Query invalidation

### Warning Split Behavior
- List-level warnings computed once by `computeListWarnings()` and rendered in ArmyListSummaryBar
- Unit-level warnings computed per-unit by `computeUnitWarnings()` and rendered inline on each ArmyListUnitRow
- "Points exceeded" hard warning removed from per-unit function, shown only in summary bar
- "Stale points" soft warning removed from per-unit function, shown only in summary bar

### Batch Confirm (Claude's discretion)
- Not included in Phase 76. Individual confirm-per-unit is sufficient for initial implementation.
- Can be added as a follow-up if user feedback indicates it is needed.

Auto-decision: Batch confirm deferred. The Sheet-based individual confirm is the safer first iteration and aligns with existing CRUD patterns in the app.
