- Icon: `<Shield className="h-3.5 w-3.5 text-muted-foreground shrink-0" />`

## Interaction Contracts

### Restore Execution Flow (complete sequence)

1. User is in `RestorePreviewDialog` after Phase 81 validation — sees "Replace current database" button
2. User clicks "Replace current database"
3. `handleConfirmRestore` sets `isRestoring = true` → dialog button shows "Restoring..." spinner, both buttons disabled
4. Frontend calls `invoke("restore_from_backup", { path })`:
   - Rust internally creates safety backup — if this fails, command returns error
   - Rust deletes sidecar files (WAL/SHM/journal)
   - Rust extracts `hobbyforge.db` from zip, writes to app_data_dir
5a. On error: `isRestoring = false`, `setPreviewOpen(false)`, `setManifest(null)`, `toast.error("Restore failed: {message}")`
5b. On success: call `relaunch()` — app restarts immediately (no toast needed, UI vanishes)

**No success toast** — the app restart is the unambiguous success signal.

### Pre-Sync Safety Backup Flow

1. User triggers Wahapedia sync
2. `useRulesSync` mutation calls `invoke("create_safety_backup")` silently
3a. Safety backup succeeds — sync continues, no UI feedback
3b. Safety backup fails — `toast.error("Safety backup failed — sync cancelled.")`, mutation rejects, sync UI resets to idle

### SafetyBackupsList Load Flow

1. Component mounts, React Query hook calls `invoke("list_safety_backups")`
2. Loading: skeleton rows (3 lines)
3a. Data arrives — list rendered, sorted newest first
3b. Empty — empty state copy shown
3c. Error — muted error message shown

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA (restore confirm) | "Replace current database" |
| Restore in-progress label | "Restoring..." |
| Restore success | (none — app relaunches) |
| Restore error — safety backup failed | "Safety backup failed — restore cancelled. {error detail}" |
| Restore error — file operation failed | "Restore failed: {error detail}" |
| Pre-sync safety backup error | "Safety backup failed — sync cancelled." |
| Safety backups list heading | "Safety Backups" |
| Safety backups empty heading | "No safety backups yet" |
| Safety backups empty body | "Safety backups are created automatically before each restore and rules sync." |
| Safety backups loading error | "Could not load safety backups." |

**Tone notes (from CONTEXT.md specifics):**
- Restore should feel "final but safe" — no hedging language, but the word "automatically" on the safety backup copy builds confidence
- Safety backup listing is "understated" — no icons, no emphasis colors in the empty state
- Pre-sync safety backup failure should be informative but terse — users understand they can retry the sync

**Destructive actions:**

| Action | Trigger | Confirmation Approach |
|--------|---------|----------------------|
| Replace current database | AlertDialogAction click in RestorePreviewDialog | AlertDialog (Phase 81 built this — no change needed). Button uses `variant="destructive"`. |

No additional confirmation dialogs in this phase — the user already passed the Phase 81 confirmation gate.
