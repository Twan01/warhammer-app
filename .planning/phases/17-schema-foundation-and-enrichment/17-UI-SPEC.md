## Component Inventory

### Surface 1: UnitDetailSheet — Details tab (ENRCH-02, ENRCH-03, ENRCH-04)

Location: `src/features/units/UnitDetailSheet.tsx`, `TabsContent value="details"` block.

**Where to insert:**

Insert after the existing "Storage Location" field row and before the existing "Linked Recipes" section (i.e., between the purchase/storage group and the recipes group). Use the existing `<Separator />` pattern to separate the new group.

```
... (existing purchase/storage fields)
<Separator />

<Field label="Undercoat">
  {unit.undercoat
    ? <span className="text-sm">{unit.undercoat}</span>
    : <span className="text-sm text-muted-foreground">—</span>
  }
</Field>

{unit.lore_notes && (
  <>
    <Separator />
    <Field label="Lore Notes">
      <p className="text-sm whitespace-pre-wrap">{unit.lore_notes}</p>
    </Field>
  </>
)}

<Separator />
... (existing Linked Recipes section)
```

Undercoat is always shown (with `—` fallback) because it is a practical painting-preparation field — users benefit from seeing it is empty. Lore Notes mirrors the existing `unit.notes` pattern: hidden when null, shown when present.

### Surface 2: UnitSheet edit form — new fields (ENRCH-02, ENRCH-03, ENRCH-04)

Location: `src/features/units/UnitSheet.tsx`, inside the expanded collapsible `<div className="flex flex-col gap-4">`.

**Where to insert:** After the existing `notes` FormField (last item in the collapsible section), append two new fields:

```
<FormField name="undercoat" ...>
  <FormLabel>Undercoat</FormLabel>
  <FormControl>
    <Input placeholder="e.g. Chaos Black, Wraithbone" {...field} value={field.value ?? ""} />
  </FormControl>
  <FormMessage />
</FormField>

<FormField name="lore_notes" ...>
  <FormLabel>Lore Notes</FormLabel>
  <FormControl>
    <textarea
      className={TEXTAREA_CLASS}
      rows={4}
      placeholder="Character history, battle honours, personal backstory…"
      {...field}
      value={field.value ?? ""}
    />
  </FormControl>
  <FormMessage />
</FormField>
```

TEXTAREA_CLASS constant: `"flex min-h-[80px] w-full rounded-md border border-input bg-background px-3 py-2 text-sm ring-offset-background placeholder:text-muted-foreground focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2 disabled:cursor-not-allowed disabled:opacity-50"` — matches existing pattern in UnitSheet notes field and PlaybookTab.

Undercoat is a single-line `<Input>` (short free-text label like "Chaos Black"). Lore Notes is a multi-line `<textarea>` with `rows={4}` (longer narrative content).

### Surface 3: FactionSheet — new field (ENRCH-01)

Location: `src/features/factions/FactionSheet.tsx`, inside the `<form>` between the existing `description` field and the `color_theme` field.

**Where to insert:** After the `description` FormField, before the `color_theme` FormField:

```
<FormField name="lore_notes" ...>
  <FormLabel>Lore Notes</FormLabel>
  <FormControl>
    <textarea
      className={TEXTAREA_CLASS}
      rows={4}
      placeholder="Chapter backstory, homeworld, custom lore, campaign history…"
      {...field}
      value={field.value ?? ""}
    />
  </FormControl>
  <FormMessage />
</FormField>
```

Use the same TEXTAREA_CLASS as the unit form. This places lore notes logically adjacent to the description field (both are narrative text fields) while keeping the color/visual fields together below.

### No new page / no new tab

Phase 17 introduces no new route, no new page component, and no new tab in UnitDetailSheet. The three existing surfaces (UnitDetailSheet Details tab for read display, UnitSheet collapsible for unit edit, FactionSheet form for faction edit) are sufficient to satisfy ENRCH-01..04.

The PlaybookTab already handles its own save cycle. The new lore/undercoat fields on units follow the UnitSheet/UnitDetailSheet save/display split already established for all other unit fields (edit in UnitSheet form → display in UnitDetailSheet Details tab).

## Interaction Contract

### UnitSheet — undercoat + lore notes fields

- Both fields live inside the existing collapsible "More details" section — no change to expand/collapse trigger or animation
- `undercoat`: shadcn `<Input>` — single-line free text, no validation constraints (any string or empty)
- `lore_notes`: raw `<textarea>` with TEXTAREA_CLASS — 4 rows, resizable by browser default
- Both fields follow the existing pattern: `value={field.value ?? ""}` and `onChange` coerces empty string to null before form submission
- Form reset behavior: both fields clear to empty string (null) when switching between create and edit modes, via the existing `form.reset()` in `useEffect([unit, defaultFactionId])` — no additional reset logic needed
- Save behavior: fields persist on "Save Unit" — the existing `useUpdateUnit` mutation payload must include `undercoat` and `lore_notes` columns

### UnitDetailSheet — Details tab display

- Read-only display: no inline editing in the Details tab for these fields (matches all other Details tab fields)
- Undercoat: always rendered below Storage Location, with `—` if null — users open UnitSheet to edit it
- Lore Notes: rendered conditionally (only if `unit.lore_notes` is truthy), using `whitespace-pre-wrap` to preserve line breaks — matches the existing `unit.notes` read-only pattern
- No "Edit" button or inline toggle on lore_notes — edit always goes through "Edit Unit" footer button

### FactionSheet — lore notes field

- Textarea is always visible in FactionSheet (not hidden behind a toggle) — the form is short enough that the extra field does not create scroll fatigue
- The existing `useEffect([faction, form])` reset must include `lore_notes` in both the faction-present branch and the DEFAULT_VALUES fallback
- Save behavior: `onSubmit` passes `lore_notes: values.lore_notes || null` to `useUpdateFaction` / `useCreateFaction` — empty string coerces to null

### Focus and keyboard

- Both new textareas are standard focusable elements — no special keyboard handling required
- Tab order follows DOM order: undercoat before lore_notes in UnitSheet, lore_notes after description in FactionSheet

## Data Binding Notes

These are implementation contracts for the executor — not design decisions, but directly implied by the UI placement.

| Field | DB column | Type | Nullable | Form coercion |
|-------|-----------|------|----------|---------------|
| `unit.undercoat` | `units.undercoat` (migration 007) | `TEXT` | yes | `""` → `null` on submit |
| `unit.lore_notes` | `units.lore_notes` (migration 007) | `TEXT` | yes | `""` → `null` on submit |
| `faction.lore_notes` | `factions.lore_notes` (migration 007) | `TEXT` | yes | `""` → `null` on submit |

Unit type (`src/types/unit.ts`) must be updated to add `undercoat: string | null` and `lore_notes: string | null`.
Faction type (`src/types/faction.ts`) must be updated to add `lore_notes: string | null`.
Unit schema (`unitSchema`) must be updated to include `undercoat` and `lore_notes` fields.
Faction schema (`factionSchema`) must be updated to include `lore_notes` field.
Query functions (`units.ts`, `factions.ts`) must SELECT and UPDATE the new columns.
