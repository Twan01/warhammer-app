  <how-to-verify>
    Run the app:
    ```
    pnpm tauri dev
    ```

    Wait for the desktop window to launch with the HobbyForge sidebar.

    **Step 1 — PINV-01: Inventory table renders.**
    1. Click "Paints" in the sidebar (Droplets icon).
    2. Confirm the page heading is "Paints" and there is an "Add Paint" button to its right.
    3. Confirm the table has 7 column headers in this exact order: Name, Brand, Type, Color Family, Owned, Recipes, Actions.
    4. Confirm at least one row renders with: a 16px color swatch left of the paint name, brand text, paint type text, color family (or `—` if null), an "Owned" or "Not owned" badge, a "Used in N recipes" or "0 recipes" badge, and Edit + Delete icon buttons.

    **Step 2 — PINV-02: Multi-select filters narrow the table.**
    1. Click the "Brand" filter button. A popover with checkboxes opens. Click "Citadel" — popover stays open, the brand button shows a count badge "1", and the table narrows to Citadel paints only.
    2. Click "Vallejo" (or another brand if Vallejo isn't seeded) — count badge becomes "2", table includes both brands.
    3. Click outside the popover to dismiss. Click "Brand" again, click "Citadel" again to deselect — count badge updates.
    4. Repeat for "Type" filter (e.g., select "Layer" then "Shade") and "Color Family" filter.
    5. Click "Clear filters" — table returns to all paints, all count badges disappear.

    **Step 3 — PINV-03: Running Low preset.**
    1. With no other filters active, click "Running Low" toggle button. The button visually flips from outline to filled (variant="default").
    2. Confirm the table now shows only paints whose `running_low = 1`. (If no seed paint has running_low set, edit one via the Edit sheet to enable running_low, save, and re-test — or add a new paint with running_low checked.)
    3. Click "Running Low" again — toggle returns to outline, table shows all paints.

    **Step 4 — PINV-04: Wishlist preset.**
    1. Click "Wishlist" toggle button. Confirm the table narrows to paints with `wishlist = 1`.
    2. Confirm Running Low and Wishlist can both be active simultaneously (the table shows paints that are BOTH running_low AND wishlist).
    3. Confirm "Clear filters" appears whenever any filter or preset is active and clears everything.

    **Step 5 — PINV-06: Inline owned toggle with optimistic update.**
    1. Pick a paint with "Not owned" badge. Click the badge. It must flip to "Owned" IMMEDIATELY (no perceptible delay — that's the optimistic update from `qc.setQueryData`).
    2. Click again — flips back to "Not owned" immediately.
    3. Navigate to Dashboard, then back to Paints — the change persists (DB write succeeded).
    4. Press Tab until focus reaches an Owned badge, then press Enter. Confirm the badge flips. Press Space — flips back. (Keyboard support for accessibility.)
    5. Optional rollback test: This requires inducing a DB error (delete a paint while the mutation is in flight, or temporarily corrupt the DB). Skip unless you suspect rollback is broken — the optimistic pattern is identical to the proven `handleToggleActive` flow in CollectionPage, so rollback failure is a low-probability regression.

    **Step 6 — PINV-05: Recipe count badge + cross-page navigation.**
    1. Confirm at least one paint has a "Used in N recipes" badge with N > 0. (Seed data ships 3 recipes referencing 6 paints — check Phase 4 seed for which paints have linkage.)
    2. Click a "Used in N recipes" badge with N > 0. Confirm the URL changes to `/recipes?paintId=X` (visible in the dev tools URL bar or via the TanStack Router devtools panel at the bottom-right of the dev window).
    3. Confirm the Recipes page shows ONLY recipes that include that paint as an ingredient.
    4. On the Recipes page, click "Clear filters". Confirm all recipes return.
    5. Navigate back to Paints and click a different paint's recipe badge — confirm the new filter applies fresh.
    6. Click a "0 recipes" badge — confirm nothing happens (the zero-count badge is non-interactive per UI-SPEC).

    **Step 7 — Edit/Delete regression check.**
    1. Click the Pencil (Edit) icon on any row. Confirm `PaintSheet` opens with the paint's fields populated. Change the brand name, click Save. Confirm the table updates.
    2. Click the Trash2 (Delete) icon on a row. Confirm `PaintDeleteDialog` opens. Click "Keep Paint" — dialog closes, no row removed. Click Delete again, confirm "Delete Paint" — row is removed. (If the paint is referenced in a recipe, the FK error toast "Cannot delete paint — it's used in a recipe step." appears.)

    **Step 8 — Filter reset on navigation.**
    1. With several filters active on /paints, navigate to Dashboard, then back to Paints. Confirm all filters are reset (table shows all paints, all toggles outline, all popovers count = 0).

    If every step passes, type "approved". If any step fails, describe what broke (which step, what you saw, what you expected). The orchestrator will route failures to `/gsd:plan-phase --gaps` for closure.
