    Create migration 028_step_progress_identity.sql following the table-rebuild pattern from migration 022 (per D-01).

    CRITICAL context for the back-fill JOIN: order_index is assigned PER-SECTION (confirmed by reading RecipeFormSheet.tsx line 378-382 which calls computeOrderIndex on sec.steps, and recipeSteps.ts line 38 which maps (s, i) => order_index: i). This means in a multi-section recipe, section A might have steps with order_index 0,1,2 and section B might also have steps with order_index 0,1,2. The back-fill must disambiguate.

    Migration structure (per D-01 through D-07):
    1. PRAGMA foreign_keys = OFF
    2. CREATE TABLE unit_recipe_step_progress_new with schema per D-04: id INTEGER PRIMARY KEY AUTOINCREMENT, assignment_id INTEGER NOT NULL REFERENCES unit_recipe_assignments(id) ON DELETE CASCADE, recipe_step_id INTEGER NOT NULL REFERENCES recipe_steps(id) ON DELETE CASCADE, completed INTEGER NOT NULL DEFAULT 0, completed_at TEXT, UNIQUE(assignment_id, recipe_step_id)
    3. INSERT INTO unit_recipe_step_progress_new (assignment_id, recipe_step_id, completed, completed_at) using a SELECT that JOINs: p (unit_recipe_step_progress) -> a (unit_recipe_assignments on a.id = p.assignment_id) -> rs (recipe_steps on rs.recipe_id = a.recipe_id AND rs.order_index = p.order_index AND COALESCE(rs.section_id, -1) matches the expected section). Since progress rows have NO section_id column, and order_index is per-section, the JOIN must use the fact that for any given recipe, the combination (recipe_id, section_id, order_index) is unique in recipe_steps. For flat recipes (section_id IS NULL), order_index alone disambiguates. For sectioned recipes, the back-fill needs to reconstruct which section each progress row belongs to.

    The safest approach: use a CTE that assigns a global_ordinal via ROW_NUMBER() OVER (PARTITION BY rs.recipe_id ORDER BY COALESCE(s.order_index, 999999), rs.order_index) to recipe_steps (joining recipe_sections s on s.id = rs.section_id), then match progress rows by their order_index against this global_ordinal. This is because the old progress system stored order_index as a flat 0..N global sequence (the UI renders steps in section-order then step-order, and progress was written using the checkbox index which was global).

    Actually, re-examining the old system: AssignmentChecklist.tsx iterates steps in section order (accordion sections, then steps within each section) and calls handleToggle(step.order_index). Since order_index is per-section (0,1,2 within each section), the old progress rows stored the per-section order_index. This means the JOIN must match (recipe_id, section_id, order_index) but progress rows have no section_id.

    Resolution: Use a numbered CTE approach. Create a CTE that numbers all recipe_steps globally per recipe (sorted by section order then step order_index), and number all existing progress rows per assignment. Then match the Nth progress row to the Nth step. BUT this is fragile if progress rows were written non-sequentially.

    Simpler correct approach: Since order_index IS per-section but the progress table has no section context, use a JOIN that matches on (recipe_id, order_index) but is tolerant of duplicates. If a recipe has duplicate order_index values across sections, take the FIRST match. Use a CTE with ROW_NUMBER to deduplicate:

    WITH numbered_steps AS (
      SELECT rs.id as step_id, a.id as assignment_id, rs.order_index,
             ROW_NUMBER() OVER (
               PARTITION BY a.id, rs.order_index
               ORDER BY COALESCE(s.order_index, 0), rs.id
             ) as rn
      FROM recipe_steps rs
      JOIN unit_recipe_assignments a ON a.recipe_id = rs.recipe_id
      LEFT JOIN recipe_sections s ON s.id = rs.section_id
    )
    INSERT INTO unit_recipe_step_progress_new (assignment_id, recipe_step_id, completed, completed_at)
    SELECT p.assignment_id, ns.step_id, p.completed, p.completed_at
    FROM unit_recipe_step_progress p
    JOIN numbered_steps ns ON ns.assignment_id = p.assignment_id AND ns.order_index = p.order_index AND ns.rn = 1

    This handles: (a) flat recipes where order_index is unique per recipe -- rn=1 always, (b) multi-section recipes where order_index collides across sections -- takes the first section's match. Per D-06, rows that cannot match are dropped (INNER JOIN).

    4. DROP TABLE unit_recipe_step_progress
    5. ALTER TABLE unit_recipe_step_progress_new RENAME TO unit_recipe_step_progress
    6. PRAGMA foreign_keys = ON

    Then register migration 028 in src-tauri/src/lib.rs get_migrations() after migration 027 entry. Add Migration { version: 28, description: "step_progress_identity", sql: include_str!("../migrations/028_step_progress_identity.sql"), kind: MigrationKind::Up }.
