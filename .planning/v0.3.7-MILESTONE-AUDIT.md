## Integration Findings

### BLOCKER: SCP-01 — RecipeFormSheet Pre-Fill Dead Code

`RecipeFormSheet` accepts `defaultFactionId` and `defaultUnitId` optional props (lines 68-69), and `buildDefaults()` correctly uses them (lines 87-96). However, **no caller passes these props**:
- `AppLayout.tsx`: `<RecipeFormSheet recipe={null} onClose={closeQuickAdd} />` — no defaultFactionId
- `RecipesPage.tsx`: `<RecipeFormSheet recipe={editing} onClose={closeForm} />` — no defaultFactionId

The 102-VERIFICATION.md acknowledges this as D-10: "UnitDetailSheet does not render RecipeFormSheet today." The component-level capability is ready but the end-to-end user flow is broken.

**Impact:** SCP-01 ("Recipe creation form pre-fills faction from FactionContext when opened from a unit context") is not achievable by any user action.

### WARNING: Override Flags Reset on Unit Edit

`UnitSheet.tsx` (lines 131-133) hardcodes all three override flags to `0` in the form payload. When a user edits any unit metadata (name, points, etc.), `updateUnit()` receives `status_assembly_override: 0`, which passes through `COALESCE(0, status_assembly_override)` → `0`, silently erasing any manual override set by `AssignmentChecklist`.

**Impact:** SAD-04 (manual-override guard) is fragile — editing a unit via UnitSheet undoes the guard.

### WARNING: Basing/Varnish Override Flags Have No UI Setter

Only `status_assembly_override` is set to `1` from UI (`AssignmentChecklist.tsx` line 76). `status_basing_override` and `status_varnished_override` have no UI trigger. The DB columns and syncDerivedStatuses guard logic exist but are inert.

**Impact:** SAD-04 override guard for basing/varnish statuses is technically present but unreachable by users.
